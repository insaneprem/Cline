01
🛠️ Custom Tools & MCP

Skills
On-Demand Specialist Knowledge

Give Cline specialist knowledge that loads only when needed — zero context bloat, pure focus.

Experimental
TL;DR Skills are modular instruction sets that load only when triggered. Your C# deployment skill doesn't burn tokens when you're writing C++ game logic. Scripts in scripts/ run for free — zero context cost.
◎What is it?
  • A Skill is a modular instruction set — a specialist role Cline steps into: ".NET API scaffolder", "Azure deployer", "C++ build optimizer".
  • Unlike rules (always-on), skills load on demand: ~100 tokens of metadata at startup, full instructions only when your request matches.
  • Progressive loading: metadata → instructions when triggered (<5k tokens) → bundled resources only when explicitly referenced.
  • A 500-line script in scripts/ costs zero context tokens — only its output enters context.
◎Why use it?
  • Without skills you re-explain your deployment process every session — or maintain a bloated always-on rules file that burns tokens on every request.
  • Skills keep Cline focused — Azure knowledge doesn't load when scaffolding a C++ template library, and vice versa.
  • Key win: Scripts run deterministically, consume zero instruction tokens, and inject only their output. All the power, none of the bloat.
🎯 Rule of thumb: One skill per domain. Keep cpp-game-engine separate from dotnet-api-scaffold. Toggle them independently and avoid context pollution.

# .NET API Scaffold

## 1. Create the Controller
- Place in Controllers/
- Follow pattern in templates/Controller.cs.tmpl
- Use repository pattern for all DB access

// Example controller structure
using Microsoft.AspNetCore.Mvc;

[ApiController]
[Route("api/[controller]")]
public class UserController : ControllerBase
{
    private readonly IUserRepository _repo;
    public UserController(IUserRepository repo) => _repo = repo;

    [HttpGet("{id}")]
    public async Task<IActionResult> GetAsync(int id)
        => Ok(await _repo.GetByIdAsync(id));
}

## 2. Register Route
- Add to Program.cs using MapControllers()
- Group by domain: auth, users, orders

## 3. Write xUnit Tests
- Create UserControllerTests.cs in Tests/
- Mock repo via IUserRepository interface — never real DB

// Example xUnit test
public class UserControllerTests
{
    [Fact]
    public async void GetAsync_ReturnsUser_WhenFound()
    {
        var mock = new Mock<IUserRepository>();
        mock.Setup(r => r.GetByIdAsync(1))
            .ReturnsAsync(new User { Id = 1 });
        var ctrl = new UserController(mock.Object);
        var result = await ctrl.GetAsync(1);
        Assert.IsType<OkObjectResult>(result);
    }
}

📋 .clinerules Context

Rules
Persistent Context, Always Enforced

Define your coding standards once, enforce them everywhere — automatically, silently, consistently.

Core Feature
TL;DR Write rules in .clinerules/ once. Cline follows them forever. Use YAML frontmatter paths: to scope rules to C++ or C# files — so the wrong rules never fire.
◎What is it?
  • Rules are Markdown files in .clinerules/ that give Cline persistent instructions on every request — defined once, followed automatically forever.
  • Also supports .cursorrules, .windsurfrules, and AGENTS.md for cross-tool compatibility.
  • Conditional rules use YAML frontmatter paths: to activate only when you're in matching files — C# rules never load during C++ engine work.
◎Why use it?
  • Without rules, every developer gets different output — one uses camelCase, another snake_case; raw SQL in controllers instead of repositories.
  • Rules enforce team-wide consistency with zero overhead — no re-explaining every session.
  • Critical insight: Loading every rule for every request wastes tokens and dilutes focus. Conditional rules fix this — C# style rules only fire when you're actually in C# code.
💡 Pro tip: Mention file paths explicitly in prompts — "Update src/Services/UserService.cs" reliably triggers path-based rules. "Update the user service" may not.

⚡ Slash-Command Automation

Workflows
Multi-Step Task Automation

Turn your 12-step release ritual into a single slash command. Type /release.md and walk away.

Slash Commands
TL;DR Workflows are Markdown files invoked with /filename.md. Mix precise XML tool calls with plain English steps. They encode your team's institutional knowledge as a reusable command — new hire types /release-prep.md and it just works.
◎What is it?
  • Workflows are Markdown files invoked with /filename.md in chat.
  • Steps can be plain English or precise XML tool syntax for guaranteed behavior — you can mix both freely.
  • The filename becomes the command: release-prep.md → /release-prep.md.
◎Why use it?
  • Multi-step processes are where engineers waste time and make mistakes — deploying means: check git, run tests, build Docker, update K8s manifests, tag, push — in the right order.
  • Workflows encode institutional knowledge as a command. New team member? They type /deploy.md and it just works.
⚡ Time saver: After completing a repetitive task, tell Cline: "Create a workflow for what we just did." Cline analyzes the conversation history and generates the workflow file automatically.
◎How to Use It — Step by Step

Scenario: A release workflow for a C++ engine + C# services platform.

Create the workflow file

🪝 Lifecycle Scripts

Hooks
Deterministic Guardrails

Inject determinism into AI. Validate, block, log, and augment at every step of Cline's workflow.

Programmatic
TL;DR Hooks run shell scripts at key lifecycle moments. A PreToolUse hook can block a bad file write before it happens. Rules describe — hooks enforce. That's the difference.
◎What is it?
  • Hooks are executable scripts that run at key moments in Cline's task lifecycle.
  • They receive JSON on stdin and return JSON on stdout with three effects: allow the operation, block it with an error, or inject context into the conversation.
  • They bring programmatic enforcement to what rules can only describe.
◎Why use it?
  • Rules are suggestions — Cline might still write a .c file in edge cases despite being told not to.
  • A PreToolUse hook catches it before it happens and blocks it with a clear error. That's the difference between guidance and enforcement.
  • Hooks also bridge existing tooling: run your linter before saves, trigger Slack at completion, inject environment-specific context for staging vs. production.
◎Hook Types Reference
Hook TypeWhen It FiresBest Used For
TaskStartNew task beginsInject project context, log analytics, check prerequisites
TaskResumeInterrupted task resumesRefresh state, check for upstream changes
TaskCancelTask is cancelledClean up temp files, notify Slack
TaskCompleteTask succeedsRun tests, trigger CI/CD, notify stakeholders
PreToolUseBefore any tool firesBlock dangerous ops, validate params — most powerful hook
PostToolUseAfter tool completesAudit logs, performance monitoring
UserPromptSubmitYou send a messageAnalytics, prompt enrichment, safety checks
PreCompactBefore context truncationArchive important conversation before removal
◎How to Use It — Step by Step

Scenario: Block .c files in a C++ project (enforce .cpp) + auto-inject project context at task start.

1
Open Hooks tab
🖱 Where to go
Scale icon → Hooks → New hook… → select PreToolUse
2
Review and edit the generated script
Click the pencil icon. macOS/Linux: extensionless file. Windows: PreToolUse.ps1
3
Write the C++ enforcement hook
PreToolUse (bash) — enforce .cpp in C++ project
#!/bin/bash
# Block .c file creation — enforce .cpp in this C++ project
INPUT=$(cat)
TOOL=$(echo "$INPUT" | jq -r '.preToolUse.tool')
FILE=$(echo "$INPUT" | jq -r '.preToolUse.parameters.path // empty')

if [[ "$TOOL" == "write_to_file" && "$FILE" == *.c && "$FILE" != *.cpp ]]; then
  echo '{"cancel":true,"errorMessage":"Use .cpp — this is a C++ project. .c files are not allowed."}'
  exit 0
fi
echo '{"cancel":false}'
TaskStart (bash) — inject project context
#!/bin/bash
# Inject .project-context into every task start
INPUT=$(cat)
WS=$(echo "$INPUT" | jq -r '.workspaceRoots[0] // empty')

if [[ -f "$WS/.project-context" ]]; then
  CTX=$(cat "$WS/.project-context")
  echo "{\"cancel\":false,\"contextModification\":\"$CTX\"}"
else
  echo '{"cancel":false}'
fi
PreToolUse (PowerShell) — enforce .cs in C# project on Windows
# Block .vb file creation — enforce .cs in this C# project
$input = [Console]::In.ReadToEnd() | ConvertFrom-Json
$tool  = $input.preToolUse.tool
$file  = $input.preToolUse.parameters.path

if ($tool -eq "write_to_file" -and $file -like "*.vb") {
  @{ cancel = $true; errorMessage = "Use .cs — this is a C# project." } |
    ConvertTo-Json
  exit 0
}
@{ cancel = $false } | ConvertTo-Json
4
Make executable
🖱 Terminal command (macOS/Linux)
chmod +x ~/.cline/hooks/PreToolUse
5
Enable and test
Toggle ON in Hooks tab. Ask Cline to create a .c file — it should be blocked with your custom message.
⚠️
Always review hook code before enabling. Hooks execute automatically and can block operations or run shell commands. Use stderr for debug output — stdout is reserved exclusively for the JSON response.
◎Best Practices for High Efficiency
▸Keep hooks fast — no blocking I/O in PreToolUse. Every hook adds latency. Save network calls for PostToolUse or TaskComplete hooks.
▸Use PostToolUse for analytics. Log tool name, success, duration. Over time you'll see exactly how Cline spends effort — and optimize.
▸Hooks + auto-approve = fast + safe. Enable auto-approve for speed, use PreToolUse hooks as the enforcement layer. Best of both worlds.
▸Layer global + project hooks. Both run when they coexist (global first, then workspace). Layer team policies with personal preferences.

🧠 Cross-Session Context

Memory Bank
Long-Term Context Across Sessions

Turn Cline from a stateless assistant into a persistent development partner that never forgets your project.

Methodology
TL;DR 3 commands: "initialize memory bank" → setup. "update memory bank" → save at session end. "follow your custom instructions" → resume. Never re-explain your architecture again.
◎What is it?
  • Memory Bank is a structured set of Markdown files in memory-bank/ that give Cline persistent context across sessions.
  • Since every conversation starts fresh, Memory Bank re-establishes full project context with one command: "follow your custom instructions."
◎Why use it?
  • After 3 days architecting a C++ game engine + C# services platform, Day 4 you open VS Code: all of that context is gone.
  • With Memory Bank: end day 3 with "update memory bank", start day 4 with "follow your custom instructions."
  • Cline reads the files, reconstructs the full picture, and picks up exactly where you left off — architecture, patterns, current focus, and all.
◎Memory Bank File Structure
.md
projectbrief.md
Foundation: core requirements, goals, scope. Created once, rarely changes.
.md
productContext.md
Why the project exists, problems solved, UX goals, target users.
.md
activeContext.md
Current focus, recent changes, next steps. Updates every session.
.md
systemPatterns.md
Architecture, design patterns, component relationships, critical paths.
.md
techContext.md
Tech stack (C++17, C# .NET 8, SQL Server), dev setup, constraints, build patterns.
.md
progress.md
What works, what's left, milestones, known issues, direction changes.
◎How to Use It — Step by Step

Scenario: Initialize Memory Bank for a game platform (C++ engine + C# matchmaking services + SQL Server).

1
Add custom instructions
Copy the Memory Bank instructions from the docs into Cline's custom instructions or a .clinerules file.
2
Initialize

Tell Cline: "initialize memory bank" — it creates memory-bank/ with all 6 core files.
3
Fill in projectbrief.md
The one file to invest upfront. Goals, tech stack (C++17, .NET 8, SQL Server), architectural constraints.
4
Add .clineignore immediately
Exclude build/, bin/, obj/, packages/. Drop context from 200k+ tokens to under 50k.
5
End every significant session
🖱 Prompt to use
"update memory bank" → Cline reviews all files and writes what changed.
6
Start every new session
🖱 Prompt to use
"follow your custom instructions" → Cline reads everything and continues seamlessly.
🧠
3 commands to memorize:
"initialize memory bank" — first-time setup
"update memory bank" — save current state at end of session
"follow your custom instructions" — resume from memory at start of session
◎Best Practices for High Efficiency
▸Keep files concise — they load every session. One page for projectbrief.md. Current-state only for activeContext.md. If a file grows past two pages, split detail into a linked doc Cline reads on demand.
▸activeContext.md is your daily driver. Update at session end: "what I just completed, what's next, decisions made." It's your developer handoff note to future-you.
▸Pair with Plan & Act mode. Start in Plan mode: Cline reads memory bank, you align on strategy. Switch to Act mode to execute. Prevents wasted effort from misaligned assumptions.
▸Pair Memory Bank + Checkpoints. Memory Bank = project knowledge. Checkpoints = code state. Roll back code freely without losing the context of why you made those decisions.

💾 Save States & Revert

Checkpoints
Fearless Iteration

Every file change is snapshotted automatically. The cost of a mistake drops to nearly zero.

Default: ON
TL;DR Checkpoints snapshot every file change in a shadow Git repo (separate from your actual git history). Auto-approve becomes safe. Make bold changes — one click to restore any point.
◎What is it?
  • Checkpoints are automatic file snapshots taken after every tool use — stored in a shadow Git repository completely separate from your actual Git history.
  • Your commits stay clean; Cline tracks everything behind the scenes.
  • After each tool use, a checkpoint appears in chat with Compare and Restore buttons — Compare opens a native diff, Restore offers three rollback strategies.
◎Why use it?
  • Without checkpoints, auto-approve is risky — Cline can make 15 file changes before you notice the direction is wrong.
  • With checkpoints, let Cline work autonomously at full speed — if it goes sideways, one click to restore. Iteration loop collapses from minutes to seconds.
  • Key insight: You're not reviewing every change — you're reviewing the outcome, with a safety net for everything in between.
◎Three Restore Strategies
Restore Files
Reverts project files. Keeps the conversation.

Use when: code broke but the discussion was productive.
Restore Task Only
Deletes messages after this point. Keeps the code.

Use when: code is good, re-ask the question differently.
Restore Files & Task
Resets both code and conversation.

Use when: complete do-over from a known-good state.

◎How to Use It — Step by Step
1
Verify checkpoints are enabled
🖱 Where to go
Cline Settings › Feature Settings › Enable Checkpoints (on by default)
2

Click "Restore" → choose the right strategy → give more specific guidance → let Cline retry.
◎Best Practices for High Efficiency
▸Checkpoints unlock auto-approve. Enable auto-approve for speed; rely on checkpoints for safety. Review outcomes, restore if needed. Faster than reviewing every change manually.
▸Use Compare before git add. Before committing, click Compare on the final checkpoint. Full diff in your native editor — the cleanest review possible.
▸Disable on massive monorepos if needed. On 100k+ file repos, checkpoint writes can slow Cline. Re-enable for write-heavy tasks, disable for read-only exploration.
▸Embrace risky experiments. Want to try a completely different C++ architecture pattern — ECS vs OOP? Let Cline go for it. Worst case: one click to restore. Low downside, high upside.

🤖 Parallel Research Agents

Subagents
Explore Without Context Bloat

Explore entire codebases simultaneously without filling the main agent's context window. Broad insight, zero bloat.

Experimental
TL;DR Subagents are read-only research agents that run in parallel. Each gets its own context window. The main agent receives synthesized reports — its context stays clean for actual work. Use before big refactors or when onboarding to unfamiliar C++/C# codebases.
◎What is it?
  • Subagents are focused research agents Cline spawns in parallel — each gets its own prompt and an independent context window.
  • Each subagent explores the codebase independently, then returns a focused report to the main agent — which receives synthesized findings, not raw file sprawl.
  • Subagents are read-only researchers — they can read, search, and run read-only commands, but cannot edit files, access MCP servers, use the browser, or spawn nested subagents.
◎Why use it?
  • Reading files sequentially in a large C++/C# codebase burns through the main agent's context window fast.
  • Three subagents explore three areas simultaneously, each with their own context budget.
  • The main agent receives three concise reports and synthesizes them — with its context window still largely intact for actual editing work.
◎Available Subagent Tools
read_file
Read full file contents from the codebase
list_files
List directory contents and structure
search_files
Regex search across files in the project
list_code_definition_names
List top-level classes, functions, methods
execute_command
Read-only: ls, grep, git log, git diff
use_skill
Load and activate skills during research
◎How to Use It — Step by Step

When sequential reading would exhaust the main context window, subagents let you explore broadly without the tradeoff.
⚠️
Skip subagents for small, focused tasks. If you already know which files to look at, subagents add unnecessary overhead and API spend. They shine for broad exploration, not targeted edits. Use command pipelines like grep -r --include=*.cpp | sort | uniq to help subagents narrow output efficiently before deep-diving.
◎Best Practices for High Efficiency
▸One focused question per subagent. "How does the C++ render pipeline work?" is a good subagent prompt. "Explore the whole codebase" is too broad — split into 3 specific research questions, one per agent.
▸Monitor per-subagent costs. Each subagent has its own token budget tracked separately. Check the cost breakdown before launching many agents on a large C++/C# codebase to avoid surprise API spend.
▸Pair with Skills. Subagents can activate skills via use_skill. A "cpp-architecture-analyzer" skill used by subagents delivers structured, consistent, domain-specific research at scale.
▸Leverage command pipelines. Tell subagents to filter first: "Use grep -r 'class.*Repository' --include=*.cs -l to find relevant files, then read the top 5." Narrow before deep-diving — faster and cheaper.
▸Use before large refactors. Before a big architectural change (e.g., migrating from raw pointers to smart pointers), launch subagents to map all affected files and dependencies. Main agent then has complete context for safe, informed edits.
