🔌 Custom Tooling
Skills & MCPs
Giving Cline New Powers
Plug in any capability your project needs — databases, build systems, issue trackers — through the open Model Context Protocol standard.
Foundation
◎ What is it?

A Skill (powered by the Model Context Protocol) is a plug-in that gives Cline a capability it doesn't have out of the box — querying a database, calling an internal build system, inspecting binary output from your compiler, or fetching tickets from an issue tracker. MCP is an open standard; tools are composable and can be written in any language.

◎ Why use it?

Cline's defaults are generic. Your toolchain isn't. Skills close that gap. Instead of you copy-pasting tool output into the chat, Cline invokes the tool directly, reads its output, and acts on it — no context switching, no copy-paste errors. Cline can chain multiple tool calls autonomously within a single task.

◎ How to use it
1
Open the MCP Servers panel inside Cline. Look for the plug icon in the top-right toolbar of the Cline panel, then click it to open the MCP Servers list.
2
Open the MCP Settings editor. Click "Edit MCP Settings" to open cline_mcp_settings.json directly in VS Code.
3
Add a server entry. Each MCP server is declared in the mcpServers object with a name, command, arguments, and any environment variables it needs.
4
Save and confirm the server appears. Return to the MCP Servers panel — a green dot means the server is running and its tools are now available to Cline in every task.
cline_mcp_settings.json
{ "mcpServers": { "project-db": { "command": "node", "args": ["/path/to/db-mcp-server/index.js"], "env": { "DB_URL": "postgresql://localhost/myapp" } }, "jira-tracker": { "command": "python", "args": ["-m", "jira_mcp"], "env": { "JIRA_TOKEN": "${env:JIRA_API_TOKEN}" } } } }
◎ Best Practices
▸
One server per concern. Keep database tools, CI tools, and issue-tracker tools in separate servers so you can enable or disable them per project.
▸
Use environment variables for secrets. Never hard-code API keys in the JSON; use ${env:VAR} syntax and store values in your shell profile.
▸
Write a brief tool description. Cline uses the tool's description to decide when to invoke it — a clear description means fewer mis-fires.
💡
Pro tip: You can ask Cline itself to help you write an MCP server. Describe the tool you want, and Cline will scaffold the boilerplate — then test it by asking Cline to use the new tool immediately.

🗄️

📋 Constraints & Personas
Rules
Shaping Cline's Behaviour
Standing instructions that persist across every task — enforce standards, codify team conventions, and give Cline a consistent engineering persona.
Core Config
◎ What is it?

Rules are standing instructions you write once and Cline applies to every subsequent task. They live in a .clinerules file at the root of your project (project-level) or in Cline's global settings (user-level). Unlike a prompt you type each time, rules are always on.

◎ Why use it?

Without rules, you repeat the same constraints in every prompt: "use TypeScript strict mode," "no magic numbers," "always write tests." Rules encode your team's standards once and enforce them silently. They also let you give Cline a persona — a senior backend engineer, a security reviewer, a documentation writer — without re-stating it every time.

◎ How to use it
1
Create .clinerules at your project root. This file takes plain text — write constraints in natural language, one per line or grouped by category.
2
Establish a coding persona first. The opening lines should describe the role Cline should adopt — this frames everything that follows.
3
Add language and style constraints. Specify the language, framework version, linting rules, naming conventions, and any forbidden patterns.
4
Commit the file to your repo. Every teammate and every CI run will now use the same rules automatically.
.clinerules
# Persona You are a senior systems engineer with deep expertise in Rust and distributed systems. Prefer correctness and clarity over cleverness. # Language & Style - Always use Rust edition 2021 - Prefer Result/Option over panics; never use .unwrap() in library code - All public APIs must have doc comments with at least one example - Use snake_case for variables, PascalCase for types # Testing - Write unit tests for every public function - Integration tests go in tests/ directory - Use property-based tests (proptest) for data transformations # Security - Never log sensitive data; use redaction helpers - Validate all external input at the boundary
◎ Rule Types
ScopeFile LocationBest For
Project.clinerules at repo rootTeam standards, framework conventions, domain knowledge
GlobalCline Settings → RulesPersonal style, preferred tools, default persona
Folder.clinerules in subdirectorySub-package specific rules (e.g. stricter rules for lib/)
⚠️
Keep rules concise. Every rule costs context-window tokens on every task. Aim for high-signal constraints — things Cline genuinely violates without them — rather than listing every possible best practice.
03
⚡ Speed vs Safety
Workflow Modes
Auto-Approve & Autonomy
Control exactly how much Cline does without asking — from fully supervised to full autopilot — and tune it per task type.
Efficiency
◎ What is it?

Cline's Auto-Approve settings let you define which actions it can take without pausing to ask for confirmation: read files, write files, run terminal commands, use browser tools. You can enable any combination, and set safe limits like a max number of auto-approved operations per task.

◎ The Approval Matrix
Action CategoryRisk LevelTypical Setting
Read files / directoriesLowAlways auto-approve
Write / edit filesMediumAuto-approve inside project root
Terminal commandsHighAuto-approve safe commands (build, test, lint)
Browser / external callsHighApprove manually
Delete operationsVery HighAlways confirm
◎ How to configure
1
Open Cline Settings → Auto-Approve. Each action category has a toggle and optional sub-filters (e.g. only auto-approve npm test and cargo build).
2
Set a per-task operation cap. The "Max auto-approved operations" field stops runaway tasks — Cline pauses and checks in after hitting the limit.
3
Create profiles for different contexts. A "safe" profile (read-only) for exploring unfamiliar code; a "build" profile for routine feature work; a "full" profile for greenfield tasks.
◎ Best Practices
▸
Start supervised, then expand. On a new project, approve everything manually until you trust the pattern of Cline's behaviour. Only then flip auto-approve on.
▸
Use Checkpoints (Ch. 06) as your safety net. When running with high auto-approval, snapshots let you rewind without losing work.
▸
Allowlist specific commands. Rather than approving all terminal commands, specify exactly which ones: cargo test, npm run lint, make build.
🧠
The mindset shift: Think of auto-approve not as "trusting AI" but as "defining the safe boundary." You're specifying the action space, and Cline operates inside it. Everything outside still requires you.
04
🪝 Automation
Hooks
Scripts That Run Automatically
Trigger shell scripts before or after any Cline action — format code, run tests, update docs, log decisions, without ever asking Cline to do it.
Automation
◎ What is it?

Hooks are shell scripts (or commands) that Cline executes automatically at defined lifecycle points: before a task starts, after each file is written, after a terminal command runs, or when a task completes. They run outside Cline's reasoning loop — they always fire, regardless of what Cline decided.

◎ Hook Event Types
PreTask
Runs before Cline starts. Good for pulling latest git, loading env vars, or checking prerequisites.
PostFileWrite
Fires after every file write. Format with Prettier/gofmt/rustfmt automatically.
PostCommand
Fires after terminal commands. Parse output, assert exit codes, or send Slack notifications.
PostTask
Runs when Cline marks a task complete. Run the full test suite, open a browser preview.
OnError
Fires if Cline errors or gets stuck. Capture diagnostic info, reset state, or page a human.
PreApproval
Runs before a manual approval gate. Present a diff summary to make reviewing faster.
◎ Configuration
cline_hooks.json
{ "hooks": { "PostFileWrite": [ { "match": "**/*.{ts,tsx}", "command": "npx prettier --write ${file}" }, { "match": "**/*.rs", "command": "rustfmt ${file}" } ], "PostTask": [ { "command": "npm test -- --passWithNoTests" } ], "PreTask": [ { "command": "git pull --rebase" } ] } }
⚡
Hooks vs Rules: Rules tell Cline how to think. Hooks control what happens around Cline's actions. Use both together — a rule to enforce test-writing, and a PostTask hook to actually run the tests.
05
🧠 Persistent Context
Memory Bank
Cline That Knows Your Project
A structured set of Markdown files Cline reads at the start of every task — giving it persistent knowledge of your architecture, decisions, and current state.
Context
◎ What is it?

The Memory Bank is a folder (typically memory-bank/ at your project root) containing Markdown files that Cline reads at the start of every task. Because LLMs have no memory between sessions, the Memory Bank is how Cline maintains continuity — it's the externalised long-term memory of your collaboration.

◎ Core Files
Required
projectbrief.md
The founding document. What the project does, who it's for, the core requirements. Everything else references back to this.
Required
activecontext.md
The current state of play: what's in progress, what changed last session, what's blocked. Updated by Cline after each task.
Core
systempatterns.md
Architecture decisions: how layers communicate, which patterns are canonical, where not to deviate.
Core
techcontext.md
The stack: language versions, key libraries, build tools, environment setup, external service contracts.
Core
progress.md
What's done, what's in progress, what's planned. A living changelog Cline updates and you can read at a glance.
Optional
custom/*.md
Add any domain-specific knowledge: API contracts, data models, deployment runbooks, regulatory constraints.
◎ How to bootstrap it
1
Ask Cline to initialise the Memory Bank. Prompt: "Read the codebase and create a memory-bank/ directory with projectbrief.md, systempatterns.md, techcontext.md, and activecontext.md." Cline will explore and write first drafts.
2
Review and correct the drafts. Cline's summaries are a starting point. Edit any misunderstandings — especially in systempatterns.md where wrong assumptions compound.
3
Add the update instruction to your Rules. Add: "At the end of every task, update activecontext.md and progress.md with what changed." Now Cline maintains its own memory.
4
Commit the memory-bank/ directory. It's part of your project. Every collaborator (human or AI) starts with the same shared understanding.
🌱
The compounding effect: A well-maintained Memory Bank means each new Cline task starts already knowing the full context. The time you invest in keeping it accurate pays back on every subsequent session — the longer the project runs, the higher the return.
06
💾 Safe Points
Checkpoints
Rollback Without Fear
Snapshot your entire workspace at any moment — files, terminal state, task context — and rewind to it if anything goes wrong.
Safety Net
◎ What is it?

Checkpoints are point-in-time snapshots of your project state — all files, terminal output, and Cline's task context — stored locally. They're Cline's equivalent of a git stash that includes everything, not just tracked files. You can create them manually or set Cline to create them automatically.

◎ When to use them
Before Risky Tasks
Any task that touches configuration, CI, infrastructure, or existing APIs. Create a checkpoint before you hit Send.
During Long Tasks
For tasks spanning many operations, auto-checkpoint every N actions so you never lose more than a small slice of work.
When Exploring
Trying a new approach or letting Cline go deep on high auto-approve? Checkpoint first and explore freely — rollback is instant.
◎ How to use them
1
Create a checkpoint manually. In the Cline panel, click the camera icon before starting a significant task. Name it descriptively — "before-auth-refactor".
2
Enable auto-checkpoints. In Cline Settings → Checkpoints, set an interval (e.g. every 5 auto-approved operations). Cline snapshots silently in the background.
3
Restore if needed. Open the Checkpoints panel, select the snapshot, click Restore. The workspace reverts completely — files, context, everything.
4
Branch from a checkpoint. Restore, then immediately create another checkpoint with a new name. Now try a different approach without losing the first.
⚠️
Checkpoints ≠ git commits. They capture everything including unstaged, untracked, and gitignored files. Use both: checkpoint for instant rollback within a work session, git for version history and collaboration.

🗺️ The Big Picture
How It All Compounds
Each of the six features is useful alone. Together, they form a system that multiplies your output — not linearly, but exponentially.
Summary
◎ The Compounding System

Think of these six features as layers. Rules define how Cline thinks. The Memory Bank gives it project-wide context. Skills & MCPs give it your toolchain. Workflow modes set how fast it operates. Hooks enforce quality gates automatically. And Checkpoints give you the safety net to let all of the above run at full speed.

The engineers who get the most from Cline aren't the ones who prompt better — they're the ones who've invested in the scaffolding. Invest once, collect returns on every task thereafter.
