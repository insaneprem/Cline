What & Why
◎What is a Skill?
  • A Skill is a folder with a SKILL.md file — a specialist role Cline steps into on demand: tizen-c-review, tizen-cpp-review, tizen-csharp-nui.
  • Unlike rules (always-on), skills activate only when your request matches their description. Everything else stays out of the way.
  • Bundled docs/ and templates/ load only when Cline needs them — keeping every session focused on exactly what you're doing.
◎Why use Skills?
  • Without skills you re-explain your review standards every session — or cram everything into a rules file that fires whether you need it or not.
  • Skills keep Cline focused — Tizen C memory rules stay out of C# NUI sessions, and vice versa.
  • One SKILL.md per domain. Toggle independently. The right knowledge loads at the right time.
🎯 Rule of thumb: one Skill per domain. tizen-c-review stays separate from tizen-csharp-nui. Cross-domain context is noise — cut it.
Anatomy of a Skill
◎What Goes Inside a Skill Directory

Every Skill is just a folder. SKILL.md is the only required file — it holds the trigger description and all review instructions. Everything else supports it.

tizen-code-review/ ├── SKILL.md # Required — trigger description + core review instructions ├── docs/ # Optional — language-specific review guides │ ├── c-native-review.md # Tizen C: memory safety, privilege APIs, error handling │ ├── cpp-review.md # C++: RAII, EFL patterns, threading rules │ └── csharp-nui-review.md # C# NUI: View lifecycle, async/await, data binding └── templates/ # Optional — review report boilerplate └── review-report.md
💡
Two required fields in SKILL.md: name (matches the folder name exactly) and description (what Cline reads to decide when to activate — make it specific and use your natural phrasing).
SKILL.md — tizen-code-review 
---
name: tizen-code-review
description: Review Tizen native C, C++, or C# NUI code for correctness,
             memory safety, privilege usage, TIDL interface validity,
             and EFL/NUI patterns. Use when asked to review, audit,
             inspect, or validate generated Tizen source code.
---

Creating your first skill
◎Step-by-Step: Your First Skill

Scenario: You're a Tizen code reviewer. You want Cline to enforce C memory safety, C++ EFL patterns, and C# NUI standards — without re-explaining the rules on every request.

1
Open the Skills panel
Click the scale icon at the bottom of the Cline panel → Skills tab → "New skill…" → name it tizen-code-review.
2
Write your SKILL.md
Add YAML frontmatter with a precise description. Include the exact phrases you'd say: "review this Tizen C file", "audit the NUI code", "check my TIDL interface". Route languages to their docs.
3
Add language-specific review docs
Create docs/c-native-review.md (memory, privilege APIs), docs/cpp-review.md (RAII, EFL), docs/csharp-nui-review.md (View lifecycle, async). Cline loads only the one relevant to the current file.
4
Test: does it trigger?
Say "review this Tizen C file for memory leaks". You should see "Activating skill: tizen-code-review". If not — add more of your natural trigger phrases to the description field.
docs/c-native-review.md — Tizen C Review 
# Tizen Native C — Review Checklist

## Memory Safety
- Every malloc() / calloc() must have a matching free() on ALL exit paths
- Prefer g_malloc() / g_free() for GLib-managed objects
- Flag unbounded strcpy(), sprintf() — require strncpy() / snprintf()

## Privilege & API Usage
- Cross-check every app_control_*(), location_*() call against tizen-manifest.xml
- http://tizen.org/privilege/network.get required for any socket call
- No direct syscall() — use Tizen CAPI wrappers only

## Error Handling
- Every Tizen CAPI call returns int — check against TIZEN_ERROR_NONE
- Use dlog_print() with correct log level; no bare printf() in production
docs/cpp-review.md — Tizen C++ / EFL Review 
# Tizen C++ / EFL — Review Checklist

## Object Lifecycle (RAII)
- Evas objects via elm_*() are parent-owned — do NOT delete them manually
- Eina containers: always call eina_list_free() with a data-free callback
- Pair every evas_object_smart_callback_add() with a matching del callback

## Threading
- UI updates MUST run on the main thread — wrap with ecore_main_loop_thread_safe_call_async()
- Flag any std::thread that touches Evas_Object directly

## TIDL Interface
- Verify all .tidl type definitions match the implementation — flag any mismatch
docs/csharp-nui-review.md — Tizen C# NUI Review 
# Tizen C# NUI — Review Checklist

## View Lifecycle
- Override Dispose(bool) for any View that holds unmanaged resources
- Animation objects: call .Stop() and .Dispose() before removing associated views
- Do NOT hold strong references to Window.Instance in long-lived objects

## Async / Await
- All async methods touching NUI must resume on the UI thread
- Flag async void (except event handlers) — require async Task

## Data Binding
- Confirm INotifyPropertyChanged on all bound ViewModels
- Two-way bindings on TextField: BindingMode.TwoWay must be explicit

## Privilege
- Match tizen-manifest.xml against all Tizen.Security.PrivacyPrivilegeManager calls

Docs, templates, and support files
◎Bundled Supporting Files — What Goes Where

Skills carry a full folder of resources alongside SKILL.md. Cline loads each file only when it's referenced — keeping every session lean.

  • docs/ — content too detailed for SKILL.md, or only relevant in specific situations. The review skill routes to the right language doc automatically. You never have to tell Cline which one to use.
  • templates/ — review report boilerplate, TIDL stubs, manifest snippets. A templates/review-report.md gives every review a consistent structure with no manual setup.
  • Keep SKILL.md under 5,000 words. Route deep content into docs/ files. SKILL.md stays fast and scannable; the rest loads only when relevant.
Put in docs/ when… Put directly in SKILL.md when…
Language-specific detail — C memory rules, C# NUI lifecycle, EFL threading patterns Universal rules that apply to all Tizen code regardless of language
Platform variants — mobile vs TV vs wearable specific guidelines Routing logic — which doc to load based on the file type under review
Deep checklists that would make SKILL.md too long to scan Trigger phrases and quick-reference steps used on every review
Where to put them
◎Project Skills (Team-Shared)
  • Store in .cline/skills/ at your project root.
  • Commit to Git. Every reviewer gets the same Tizen standards. When someone tightens the C# checklist, everyone picks it up on the next pull.
  • Also supported: .clinerules/skills/ and .claude/skills/.
◎Global Skills (Personal)
  • macOS/Linux: ~/.cline/skills/
  • Windows: C:\Users\USERNAME\.cline\skills\
  • Available across every project — for personal review preferences that don't belong in the shared repo.
  • Note: when a global and project skill share the same name, the global skill wins.
🔁
Toggle, don't delete. Every skill has an on/off switch. Disable the C++ review skill during a pure C# sprint — re-enable it when you need it.
Making skills actually work well
◎Best Practices — Learned the Hard Way
▸
Nail the description first. "Review Tizen C or C++ code for memory leaks, privilege misuse, or TIDL errors" fires every time. "Tizen helper" never fires. Add the exact phrases you'd naturally say.
▸
Route, don't cram. SKILL.md routes to the right doc — it doesn't contain every rule itself. docs/c-native-review.md handles C. docs/csharp-nui-review.md handles C#. Cline reads only what matches.
▸
One skill = one domain. Keep tizen-c-review and tizen-csharp-nui separate. Toggle them independently. When a request needs both, both load — otherwise they stay silent.
▸
Front-load the common cases. Cline reads SKILL.md top to bottom. Put the C native path first (most common Tizen path). Wearable-specific edge cases go last — or move them to a docs/ file.
▸
Be concrete, not abstract. "Check tizen-manifest.xml for every privilege the code calls" beats "check privileges." The more specific, the more reliably Cline follows it.
▸
Commit to Git. Store in .cline/skills/ and push. Every reviewer inherits the same standards. No meeting required — just a pull.
