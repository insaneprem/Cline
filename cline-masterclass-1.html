01
Cline Skills

Custom Tools & MCP Extensions

Give Cline specialist knowledge that loads only when needed — zero context bloat, pure focus.

Experimental
What is it?

A Skill is a modular instruction set packaged as a folder with a SKILL.md file. Each skill is a specialist role Cline steps into — "AWS deployer", "Go API scaffolder", "data analyst". Unlike rules (always-on), skills load on demand: metadata (~100 tokens each) is available at startup, and the full instructions fire only when your request matches the description.

Skills use progressive loading across three levels: metadata at startup, instructions when triggered (<5k tokens), and bundled resources only when explicitly referenced. A 500-line validation script in scripts/ costs zero context tokens — only its output does.

📸
Screenshot: Cline Skills Panel in VS Code
The Cline sidebar showing the Skills tab (scale icon, bottom-left). Three skills with toggle switches: "go-api-scaffold" (ON), "aws-cdk-deploy" (ON), "data-analysis" (OFF). Each shows name, description excerpt, and last-modified date.
Why Use It?

Without skills you either re-explain your deployment process every session, or maintain a massive always-on rules file that burns context on every Go function. Skills keep Cline focused — AWS knowledge doesn't load when you're scaffolding a C++ template, and vice versa.

Key win: Scripts in scripts/ run deterministically, consume zero instruction tokens, and inject only their output into context. All the power, none of the bloat.

How to Use It — Step by Step

Scenario: Go microservice API — scaffold handlers, routes, and tests consistently every time.

  1. 01
    Enable Skills: Settings → Features → Enable Skills
  2. 02
    Open Skills tab: Scale icon → "Skills" → "New skill…" → name it go-api-scaffold
  3. 03
    Edit the SKILL.md with a focused description and step-by-step instructions. Put the most common cases first — Cline reads sequentially.
  4. 04
    Add supporting files: Create templates/handler.go.tmpl and scripts/validate-routes.sh for zero-cost deterministic operations.
  5. 05
    Test the trigger: Say "scaffold a new user auth endpoint" — if the skill fires you'll see "Activating skill: go-api-scaffold" in Cline output.
SKILL.md
--- name: go-api-scaffold description: Scaffold Go REST API handlers, routes, middleware, and table-driven tests. Use when creating new API endpoints, adding routes, or generating handler boilerplate in a Go microservice. --- # Go API Scaffold ## 1. Create the Handler - Place in internal/handlers/ - Follow pattern in templates/handler.go.tmpl - Repository pattern for all DB access ## 2. Register Route - Add to internal/router/router.go - Group by domain: auth, users, orders ## 3. Write Table-Driven Tests - Create _test.go alongside the handler - Mock DB via interfaces, never real DB in unit tests ## Validate after scaffolding: bash scripts/validate-routes.sh
Best Practices for High Efficiency
  • 🎯
    Nail the description — it's the triggerBe specific: "Use when deploying to AWS using CDK, updating infrastructure stacks, or managing AWS resources." Vague descriptions mean the skill never fires.
  • 📁
    Offload weight to scripts/Complex logic in scripts/ costs zero tokens. A 300-line linter becomes a single "Passed" or structured error in context.
  • 🔀
    One skill = one domainKeep go-api-scaffold separate from go-db-migration. Toggle them independently. Domain isolation prevents context pollution.
  • 🗂️
    Commit project skills to GitStore in .cline/skills/ and commit. Your whole team benefits — shared C++ competitive programming templates mean everyone scaffolds identically.

Cline Rules

Persistent Context with .clinerules

Define your coding standards once, enforce them everywhere — automatically, silently, consistently.

Core Feature
What is it?

Rules are markdown files that give Cline persistent instructions on every request. Define them once in .clinerules/ and Cline follows them automatically — no re-explaining each session. Cline also supports .cursorrules, .windsurfrules, and AGENTS.md for cross-tool compatibility.

Conditional rules use YAML frontmatter paths: to activate only when you're working in matching files. Your React rules don't load during Go backend work. Focused context → faster, more accurate responses and lower token costs.

📸
Screenshot: Rules Panel with Conditional Scopes
Cline Rules tab showing 4 rule files: universal.md (ON, no scope), go-backend.md (ON, conditional — paths: services/**, internal/**), nextjs-frontend.md (ON, conditional — paths: app/**, components/**), legacy-freeze.md (OFF). Each shows toggle + scope indicator.
Why Use It?

Without rules, every developer using Cline on your Go + Next.js platform gets different output — one uses err, another uses e; one writes raw SQL in handlers, another uses repositories. Rules enforce team-wide consistency without any overhead.

The critical insight: loading every rule for every request wastes tokens and dilutes focus. Conditional rules solve this — frontend style rules compete for Cline's attention only when you're actually in frontend code.

How to Use It — Step by Step

Scenario: Go backend microservice + Next.js frontend with distinct coding standards for each layer.

  1. 01
    Create: .clinerules/ at project root.
  2. 02
    Open Rules panel: Scale icon → "Rules" → "New rule file…" → name it go-backend
  3. 03
    Write the conditional Go rule with YAML frontmatter:
.clinerules/go-backend.md
--- paths: - "services/**" - "internal/**" - "cmd/**" --- # Go Backend Guidelines ## Naming - Variables: camelCase (userID not user_id) - Exported: PascalCase only - Errors: always err, wrapped with fmt.Errorf("context: %w", err) ## Architecture - Repository pattern for all DB access - Never raw SQL in handlers — use internal/store/ - See error pattern: internal/errors/errors.go - DO NOT modify /legacy (Q2 removal scheduled) ## Testing - Table-driven tests for all public functions - Mock DB via interfaces, never real DB in unit tests
  1. 04
    Add frontend rule: .clinerules/nextjs-frontend.md with paths: ["app/**", "components/**"]
  2. 05
    Add universal rule: .clinerules/universal.md with no frontmatter — applies to every request.
  3. 06
    Verify: Open a file in services/ → look for "Conditional rules applied: workspace:go-backend.md" notification.
💡
Tip: Mention file paths explicitly in prompts — "Update src/services/user.ts" reliably triggers path-based rules. "Update the user service" may not.
Best Practices for High Efficiency
  • ✂️
    Specific over vague — always"camelCase for Go, UPPER_SNAKE for C++ constants" beats "use descriptive names." Concrete rules leave no room for drift.
  • 💡
    Include the "why""Don't modify /legacy (Q2 removal)" helps Cline make smarter edge-case decisions. Context = better judgment.
  • 📌
    Point to existing code examples"Follow the pattern in internal/errors/errors.go" is 10× more effective than describing it in prose. Cline reads that file directly.
  • 🗂️
    One file per concernSeparate coding.md, testing.md, architecture.md. Disable strict testing while prototyping, re-enable before review.

Cline Workflows

Multi-Step Task Automation

Turn your 12-step release ritual into a single slash command. Type /release.md and walk away.

Slash Commands
What is it?

Workflows are Markdown files invoked with /filename.md in chat. Steps can be plain English (Cline decides implementation) or precise XML tool syntax for guaranteed behavior — you can mix both. The filename becomes the command: release-prep.md → /release-prep.md.

📸
Screenshot: Workflow Slash-Command Autocomplete
Cline chat input with "/" typed, showing a dropdown: deploy.md, release-prep.md, setup-db.md, test-and-commit.md — each with description. User hovering over "release-prep.md". Step execution visible in the conversation above with approval gates.
Why Use It?

Multi-step processes are where engineers waste time and make mistakes. Deploying a microservice means: check git status, run tests, build Docker images, update K8s manifests, tag, push — in the right order. Workflows encode institutional knowledge as a command. New team member? They type /deploy.md and it just works.

How to Use It — Step by Step

Scenario: A release workflow for a Go + Next.js microservices platform.

  1. 01
    Create: Scale icon → Workflows tab → "New workflow file…" → name it release-prep
  2. 02
    Write the workflow file (XML for precision, prose for flexibility):
.clinerules/workflows/release-prep.md
# Release Preparation Prepare a release: run tests, bump version, write changelog. ## Step 1: Verify clean working directory <execute_command> <command>git status --porcelain</command> </execute_command> If uncommitted changes exist, ask whether to stash or abort. ## Step 2: Run all tests <execute_command> <command>go test ./... && cd frontend && npm test -- --watchAll=false</command> </execute_command> If any test fails, show failures and STOP. ## Step 3: Choose version bump <ask_followup_question> <question>What version bump?</question> <options>["Patch (x.x.X)","Minor (x.X.0)","Major (X.0.0)"]</options> </ask_followup_question> ## Step 4: Update version files Update version.go and package.json to the new version. ## Step 5: Generate changelog <execute_command> <command>git log --oneline $(git describe --tags --abbrev=0)..HEAD</command> </execute_command> Write a CHANGELOG.md entry from these commits.
  1. 03
    Commit: Store in .clinerules/workflows/ — team-wide instantly.
  2. 04
    Invoke: Type /release-prep.md in Cline chat. Cline executes each step, pausing for approval at the version question.
⚡
Power tip: After completing a repetitive task, tell Cline: "Create a workflow for what we just did." Cline analyzes the conversation history and generates the workflow file automatically.
Best Practices for High Efficiency
  • 🛡️
    Always define failure behaviorEvery failable step must say what happens: "If tests fail, show failures and stop." Don't let Cline blunder forward after a broken build.
  • 🔗
    Compose, don't bloatKeep deploy.md focused on deploying. Chain separate workflows in chat rather than one mega-workflow that does everything.
  • 🔒
    Review before runningWorkflows execute with your full permissions. Always read a workflow from an external source before invoking it for the first time.

Cline Hooks

Lifecycle Scripts & Guardrails

Inject determinism into AI. Validate, block, log, and augment at every step of Cline's workflow.

Programmatic
What is it?

Hooks are executable scripts that run at key moments in Cline's task lifecycle. They receive JSON on stdin, return JSON on stdout with three possible effects: allow the operation, block it with an error message, or inject context into the conversation. They bring programmatic enforcement to what rules can only describe.

Hook TypeWhen It FiresBest Used For
TaskStartNew task beginsInject project context, log analytics, check prerequisites
TaskResumeInterrupted task resumesRefresh state, check for upstream changes
TaskCancelTask is cancelledClean up temp files, notify Slack
TaskCompleteTask succeedsRun tests, trigger CI/CD, notify stakeholders
PreToolUseBefore any tool firesBlock dangerous ops, validate params — most powerful hook
PostToolUseAfter tool completesAudit logs, performance monitoring
UserPromptSubmitYou send a messageAnalytics, prompt enrichment, safety checks
PreCompactBefore context truncationArchive important conversation before removal
Why Use It?

You've told Cline "always use TypeScript" in a rule. Rules are suggestions — Cline might still write a .js file in edge cases. A PreToolUse hook catches it before it happens and blocks it with a clear error. That's the difference between guidance and enforcement.

Hooks also bridge your existing tooling: run your linter before files are saved, trigger Slack at task completion, inject environment-specific context when working on staging vs. production services.

How to Use It — Step by Step

Scenario: Block .js files in a TypeScript project + auto-inject project context at task start.

  1. 01
    Open Hooks tab: Scale icon → "Hooks" → "New hook…" → select PreToolUse
  2. 02
    Review and edit the generated script (pencil icon). macOS/Linux: extensionless file. Windows: PreToolUse.ps1
  3. 03
    Write the TypeScript enforcement hook:
PreToolUse (bash)
#!/bin/bash # Block .js file creation in this TypeScript project INPUT=$(cat) TOOL=$(echo "$INPUT" | jq -r '.preToolUse.tool') FILE=$(echo "$INPUT" | jq -r '.preToolUse.parameters.path // empty') if [[ "$TOOL" == "write_to_file" && "$FILE" == *.js ]]; then echo '{"cancel":true,"errorMessage":"Use .ts — TypeScript project"}' exit 0 fi echo '{"cancel":false}'
TaskStart (bash) — context injection
#!/bin/bash # Inject .project-context into every task start INPUT=$(cat) WS=$(echo "$INPUT" | jq -r '.workspaceRoots[0] // empty') if [[ -f "$WS/.project-context" ]]; then CTX=$(cat "$WS/.project-context") echo "{\"cancel\":false,\"contextModification\":\"$CTX\"}" else echo '{"cancel":false}' fi
  1. 04
    Make executable: chmod +x ~/.cline/hooks/PreToolUse (macOS/Linux)
  2. 05
    Enable: Toggle ON in Hooks tab. Test by asking Cline to create a .js file — it should be blocked with your custom message.
⚠️
Always review hook code before enabling. Hooks execute automatically and can block operations or run shell commands. Use stderr (>&2) for debug output — stdout is reserved exclusively for the JSON response.
Best Practices for High Efficiency
  • ⚡
    Keep hooks fast — no blocking I/O in PreToolUseEvery hook adds latency. Save network calls for PostToolUse or TaskComplete hooks.
  • 📊
    Use PostToolUse for analyticsLog tool name, success, duration. Over time you'll see exactly how Cline spends effort — and optimize.
  • 🔐
    Hooks + auto-approve = fast + safeEnable auto-approve for speed, use PreToolUse hooks as the enforcement layer. Best of both worlds.
  • 🗂️
    Layer global + project hooksBoth run when they coexist (global first, then workspace). Layer team policies with personal preferences.

Cline Memory Bank

Long-Term Context Across Sessions

Turn Cline from a stateless assistant into a persistent development partner that never forgets your project.

Methodology
What is it?

Memory Bank is a documentation methodology — a structured set of Markdown files in memory-bank/ that give Cline persistent context across sessions. Since every conversation starts fresh, Memory Bank re-establishes full project context with one command: "follow your custom instructions."

.md
projectbrief.md
Foundation: core requirements, goals, scope. Created once, rarely changes.
.md
productContext.md
Why the project exists, problems solved, UX goals, target users.
.md
activeContext.md
Current focus, recent changes, next steps. Updates most frequently — every session.
.md
systemPatterns.md
Architecture, design patterns, component relationships, critical paths.
.md
techContext.md
Tech stack (Go, Next.js, Postgres), dev setup, constraints, tool patterns.
.md
progress.md
What works, what's left, milestones, known issues, direction changes.
Why Use It?

You spend 3 days architecting a Go microservices platform — CQRS here, event sourcing there, specific error patterns everywhere. Day 4 you open VS Code and start a new Cline task: all of that context is gone. You spend 30 minutes re-orienting Cline before any work happens.

With Memory Bank: end day 3 with "update memory bank", start day 4 with "follow your custom instructions." Cline reads the files, reconstructs the full picture, and picks up exactly where you left off — architecture, patterns, current focus, and all.

How to Use It — Step by Step

Scenario: Initialize Memory Bank for a microservices blog platform (Go + Next.js + Postgres).

  1. 01
    Add custom instructions: Copy the Memory Bank instructions from the docs into Cline's custom instructions or a .clinerules file.
  2. 02
    Initialize: Tell Cline "initialize memory bank" — it creates memory-bank/ with all 6 core files.
  3. 03
    Fill in projectbrief.md: The one file to invest upfront. Goals, tech stack, architectural constraints.
  4. 04
    Add .clineignore immediately: Exclude node_modules/, vendor/, build artifacts. Drop context from 200k+ tokens to under 50k.
  5. 05
    End every significant session: "update memory bank" → Cline reviews all files and writes what changed.
  6. 06
    Start every new session: "follow your custom instructions" → Cline reads everything and continues seamlessly.
🧠
3 commands to memorize:
"initialize memory bank" — first-time setup
"update memory bank" — save current state
"follow your custom instructions" — resume from memory
Best Practices for High Efficiency
  • 📏
    Keep files concise — they load every sessionOne page for projectbrief.md. Current-state only for activeContext.md. If a file grows past two pages, split detail into a linked doc Cline reads on demand.
  • 🔄
    activeContext.md is your daily driverUpdate at session end: "what I just completed, what's next, decisions made." It's your developer handoff note to future-you.
  • 🎯
    Pair with Plan & Act modeStart in Plan mode: Cline reads memory bank, you align on strategy. Switch to Act mode to execute. Prevents wasted effort from misaligned assumptions.
  • 🔗
    Pair Memory Bank + CheckpointsMemory Bank = project knowledge. Checkpoints = code state. Roll back code freely without losing the context of why you made those decisions.

Cline Checkpoints

Save States & Fearless Iteration

Every file change is snapshotted automatically. The cost of a mistake drops to nearly zero.

Default: ON
What is it?

Checkpoints are automatic file snapshots taken after every tool use — stored in a shadow Git repository completely separate from your actual Git history. Your commits stay clean; Cline tracks everything behind the scenes.

After each tool use, a checkpoint indicator appears in chat with Compare and Restore buttons. Compare opens a native diff view. Restore gives you three distinct rollback strategies.

Restore Files
Reverts project files. Keeps the conversation.

Use when: code broke but the discussion was productive.
Restore Task Only
Deletes messages after this point. Keeps the code.

Use when: code is good, re-ask the question differently.
Restore Files & Task
Resets both code and conversation.

Use when: complete do-over from a known-good state.
📸
Screenshot: Checkpoint Indicators in Cline Chat
A Cline conversation with 3 file edits. After each, a bookmark icon labeled "Checkpoint" appears with dotted connector and "Compare" / "Restore" buttons. The Restore dropdown is open on checkpoint 2, showing the 3 restore options.
Why Use It?

Without checkpoints, auto-approve is risky — Cline can make 15 file changes before you notice the direction is wrong. With checkpoints, you let Cline work autonomously at full speed. If it goes sideways: one click to restore. The iteration loop collapses from minutes to seconds.

Key insight: Checkpoints make AI-assisted coding viable at speed. You're not reviewing every change — you're reviewing the outcome, with a safety net for everything in between.

How to Use It — Step by Step
  1. 01
    Verify: Cline Settings → Feature Settings → "Enable Checkpoints" (on by default).
  2. 02
    Enable auto-approve for file edits — checkpoints make this safe. Let Cline work at full speed.
  3. 03
    Let Cline run. Watch checkpoint markers appear after each tool use in the conversation.
  4. 04
    Click "Compare" on any checkpoint to see a full diff of all changed files in your native editor diff viewer.
  5. 05
    If something's wrong: Click "Restore" → choose the right strategy → give more specific guidance → let Cline retry.
Best Practices for High Efficiency
  • 🚀
    Checkpoints unlock auto-approveThis is the key insight. Enable auto-approve for speed; rely on checkpoints for safety. Review outcomes, restore if needed. Faster than reviewing every change manually.
  • 🔬
    Use Compare before git addBefore committing, click Compare on the final checkpoint. Full diff in your native editor — the cleanest review possible.
  • ⚡
    Disable on massive monorepos if neededOn 100k+ file repos, checkpoint writes can slow Cline. Re-enable for write-heavy tasks, disable for read-only exploration.
  • 🧪
    Embrace risky experimentsWant to try a completely different architecture? Let Cline go for it. Worst case: one click to restore. Low downside, high upside.

Cline Subagents

Parallel Research Agents

Explore entire codebases simultaneously without filling the main agent's context window. Broad insight, zero bloat.

Experimental
What is it?

Subagents let Cline spawn focused research agents that run in parallel. Each subagent gets its own prompt and an independent context window, explores the codebase independently, and returns a focused report to the main agent. The main agent's context window stays clean — it receives synthesized findings, not raw file contents sprawled across its budget.

Subagents are read-only researchers. They can read, search, and run read-only commands — but cannot edit files, access MCP servers, use the browser, or spawn nested subagents. Each subagent tracks its own token and cost usage, rolled into the task total.

read_file
Read full file contents from the codebase
list_files
List directory contents and structure
search_files
Regex search across files in the project
list_code_definition_names
List top-level classes, functions, methods
execute_command
Read-only: ls, grep, git log, git diff
use_skill
Load and activate skills during research
📸
Screenshot: Parallel Subagents Running in Cline
Cline chat showing 3 subagents launched simultaneously, each with a progress indicator: "Subagent 1: Exploring auth routes (14 tool calls, ~8k tokens, $0.03)", "Subagent 2: Mapping DB models (9 tool calls, ~5k tokens, $0.02)", "Subagent 3: Tracing deployment config (11 tool calls, ~6k tokens, $0.02)". Each shows live tool call count.
Why Use It?

The problem with large codebases: reading files sequentially burns through the main agent's context window fast. By the time you've explored authentication, DB models, and deployment config, you've consumed most of your budget before writing a single line of code.

Subagents solve this with parallelism. Three agents explore three areas simultaneously, each with their own context budget. The main agent receives three concise reports and synthesizes them into a focused implementation plan — with its context window still largely intact for actual work.

How to Use It — Step by Step

Scenario: You've inherited a large Go microservices codebase. Before making changes, you want a full architectural map.

  1. 01
    Enable Subagents: Cline Settings → Features → Agent section → toggle Subagents ON. (Disabled by default — experimental.)
  2. 02
    Auto-approve (optional): If "Read project files" is enabled in Auto Approve, subagent launches are auto-approved. In YOLO mode: always auto-approved.
  3. 03
    Invoke with an explicit subagent prompt:
Example Subagent Prompts
# Onboarding to an unfamiliar codebase "Use subagents to map out the architecture, main entry points, and data flow patterns in this Go microservices project. Run them in parallel." # Pre-edit research before a major change "Before we modify the auth system, spin up subagents to investigate: (1) how authentication currently works, (2) where the user DB models are defined, and (3) how the middleware is structured." # Cross-cutting concern analysis "Use subagents to simultaneously trace how error handling, logging, and observability are implemented across all services — I want to standardize all three."
  1. 04
    Review the approval prompt (if auto-approve is off). Cline shows you the exact prompts it will send to each subagent before launching.
  2. 05
    Monitor parallel execution. Each subagent shows tool call count, token usage, and cost in real time in the Cline UI.
  3. 06
    Review the synthesized report. The main agent combines all findings into a coherent response — architecture map, key file paths, and recommended next steps.
💡
Subagents need explicit invocation. Cline does not automatically use subagents — you must ask for them. Mention "subagents" or describe parallel exploration in your prompt. Each subagent prompt should cover a focused, distinct research area.
When to Use Subagents
🗺️
Onboarding to Unfamiliar Code
Map architecture, entry points, and data flow in parallel — complete picture in one shot, not sequential exploration.
🔀
Cross-Cutting Concern Analysis
Trace auth, logging, and error handling simultaneously across all services. See system-wide patterns at once.
🔬
Pre-Edit Research
Before major changes, gather context from all related files. Main agent makes informed edits without burning context on exploration.
🏗️
Large Codebase Navigation
When sequential reading would exhaust the main context window, subagents let you explore broadly without the tradeoff.
⚠️
Skip subagents for small, focused tasks. If you already know which files to look at, subagents add unnecessary overhead and API spend. They shine for broad exploration, not targeted edits. Use command pipelines like rg ... | sort | uniq to help subagents narrow output efficiently before deep-diving.
Best Practices for High Efficiency
  • 🎯
    One focused question per subagent"How does authentication work?" is a good subagent prompt. "Explore the whole codebase" is too broad — split into 3 specific research questions, one per agent.
  • 💰
    Monitor per-subagent costsEach subagent has its own token budget tracked separately. Check the cost breakdown before launching many agents on a large codebase to avoid surprise API spend.
  • 🔗
    Pair with SkillsSubagents can activate skills via use_skill. A "go-architecture-analyzer" skill used by subagents delivers structured, consistent, domain-specific research at scale.
  • 🔍
    Leverage command pipelinesTell subagents to filter first: "Use rg --type go -l 'repository' to find relevant files, then read the top 5." Narrow before deep-diving — faster and cheaper.
  • 📋
    Use before large refactorsBefore a big architectural change, launch subagents to map all affected files and dependencies. Main agent then has complete context for safe, informed edits without consuming its context window on exploration.
