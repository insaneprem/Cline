Skills On-Demand Specialist Knowledge

Think of Skills as specialist hats Cline can wear — put one on when you need it, take it off when you don't. A deployment skill stays dormant while you're writing game logic. A scaffold skill activates the moment you ask for it. No token bloat. Pure focus.

On-Demand Loading Zero Context Bloat SKILL.md Based .cline/skills/ Git-Friendly
⚗️  Experimental Feature — Enable in Settings → Features → Enable Skills
TL;DR Skills are modular instruction sets that activate only when your request matches their description. At startup, Cline reads ~100 tokens of metadata per skill. Full instructions only load when triggered. Anything you put in scripts/ costs zero context tokens — only the output comes back.
Understanding the concept
◎So what exactly is a Skill?
  • A Skill is a directory containing a SKILL.md file — a specialist role Cline steps into: "dotnet-api-scaffold", "aws-cdk-deploy", "data-analysis".
  • Unlike rules (which are always-on), skills are lazy — metadata loads at startup (~100 tokens each), but full instructions only come in when your request triggers them.
  • Progressive loading in 3 levels: metadata → instructions (under 5k tokens) → bundled files only when you reference them.
  • That 500-line validation script sitting in scripts/? Zero context tokens. Only its output ("Passed ✓") enters the conversation.
◎Honestly, why bother?
  • Without Skills, you're either re-explaining your deployment workflow every single session — or you have one giant bloated rules file burning tokens on requests that don't need it.
  • Skills keep Cline sharp — Azure deployment knowledge doesn't sneak into your C++ template scaffolding, and vice versa. Context stays relevant.
  • The real win with scripts: deterministic operations run outside the LLM entirely. Your linter runs in bash, produces clean output, and Cline acts on results without wasting tokens on the code itself.
🎯 Simple rule of thumb: one Skill per domain. Keep cpp-game-engine separate from dotnet-api-scaffold. Toggle them independently. Cross-domain context is noise — eliminate it.
How loading actually works
◎Progressive Loading — Three Levels

Cline doesn't dump everything into context at once. It reads only what's needed, when it's needed — that's the entire point.

Level When It Loads Token Cost What's Included
Metadata Every startup, always ~100 tokens per skill name and description from YAML frontmatter — just enough for Cline to know the skill exists
Instructions When your request matches the description Under 5k tokens Full SKILL.md body — the actual step-by-step guidance and patterns
Resources Only when explicitly referenced Effectively unlimited Bundled files in docs/, templates in templates/, script outputs from scripts/
Anatomy of a Skill
◎What Goes Inside a Skill Directory

Every Skill is just a folder. The only required piece is SKILL.md — everything else is optional but powerful.

my-skill/ ├── SKILL.md # Required — your instructions + YAML frontmatter ├── docs/ # Optional — advanced guides, platform-specific notes │ └── advanced.md ├── templates/ # Optional — boilerplate files Cline can scaffold from │ └── config.yaml └── scripts/ # Optional — deterministic ops, zero context cost └── validate.sh
💡
The two required YAML fields inside every SKILL.md: name (must match the directory name exactly) and description (max 1,024 characters — this is what Cline reads to decide whether to activate the skill).
SKILL.md — structure overview 
---
name: my-skill              # must match the directory name exactly
description: Brief, specific description of what this does and WHEN to use it.
             Max 1,024 characters. This is your trigger phrase.
---

# My Skill

Detailed instructions that Cline follows when this skill is activated.
Put the most common cases first — Cline reads sequentially.

## Steps
1. First, do this
2. Then do that
3. For edge cases, see [advanced.md](docs/advanced.md)
Creating your first skill
◎Step-by-Step: Your First Skill

Scenario: You want Cline to scaffold C# ASP.NET Core controllers consistently every time — same patterns, same structure, no re-explaining.

1
Open the Skills panel
Click the scale icon at the bottom of the Cline panel (left of the model selector) → switch to the Skills tab → click "New skill…" and give it a name like dotnet-api-scaffold.
2
Write your SKILL.md
Edit the generated file. Be specific in the description — this is literally what Cline pattern-matches against your messages. Lead with the most common use cases; Cline reads the file top to bottom.
3
Add supporting files (optional but powerful)
Drop a templates/Controller.cs.tmpl for boilerplate and a scripts/validate-routes.sh for zero-cost route validation. Cline will reference these only when needed.
4
Test it — does it actually trigger?
Say "scaffold a new UserAuth API endpoint". If you see "Activating skill: dotnet-api-scaffold" in the Cline output, you nailed the description. If not, make it more specific — add exact trigger phrases you'd naturally say.
SKILL.md — dotnet-api-scaffold (real example) 
---
name: dotnet-api-scaffold
description: Scaffold C# ASP.NET Core controllers, routes, middleware, and xUnit tests.
Use when creating new API endpoints, adding routes, or generating controller
boilerplate in a .NET microservice. Triggers on: "scaffold endpoint", "new
controller", "add API route", "generate controller".
---

# .NET API Scaffold

## 1. Create the Controller
- Place in Controllers/ — follow templates/Controller.cs.tmpl
- Use repository pattern for all DB access (never hit DB directly from controller)

## 2. Register Route
- Add to Program.cs via MapControllers()
- Group by domain: auth, users, orders

## 3. Write xUnit Tests
- Create UserControllerTests.cs in Tests/
- Mock repo via interface — never use a real DB in unit tests

## 4. Validate
bash scripts/validate-routes.sh    # zero context cost — just output enters chat
✅
You can also skip the UI entirely — just create the directory structure manually and place it in .cline/skills/. Cline auto-discovers it. Either way works.
Naming your skills
◎Naming Conventions — Kebab-Case, Always Descriptive

The name you give a skill must match the directory name exactly. It also shows in Cline's UI and error logs, so make it readable at a glance. Always lowercase, always hyphens — no exceptions.

✓ Good names
  • aws-cdk-deploy
  • pr-review-checklist
  • database-migration
  • api-client-generator
  • dotnet-api-scaffold
✗ Avoid these patterns
  • aws  (too vague)
  • my_skill  (underscores ≠ kebab-case)
  • DeployToAWS  (not kebab-case)
  • misc-helpers  (too generic)
  • mySkill  (camelCase is wrong here)
The most important field
◎The Description is Your Trigger
  • Cline uses the description to decide whether to activate a skill. A vague description means the skill never fires when you need it.
  • Good descriptions start with action verbs, include specific trigger phrases you'd naturally type, and mention the tools or file types involved.
  • Max 1,024 characters — be specific but don't ramble. Think: "what would I type to need this?"
  • Test it: try different phrasings of requests and watch whether the skill activates. If it doesn't, add more trigger phrases.
◎Strong vs Weak Descriptions
  • ✅ Strong: "Deploy applications to AWS using CDK. Use when deploying, updating infrastructure, or managing AWS resources."
  • ✅ Strong: "Generate release notes from git commits. Use when preparing releases, writing changelogs, or summarizing changes."
  • ❌ Weak: "Helps with AWS stuff." — no action verb, no trigger phrase.
  • ❌ Weak: "Data analysis helper." — zero specificity. Cline won't know when to fire this.
Scripts, templates, and docs
◎Bundled Supporting Files — What Goes Where

Skills aren't just a single file — they can carry a whole folder of supporting resources. Cline loads these only when referenced, keeping context lean.

  • docs/ — for information too detailed to fit in SKILL.md, or only relevant in specific situations. A deploy skill might have docs/aws.md, docs/gcp.md, and docs/azure.md. Cline picks the right one based on your request.
  • templates/ — for boilerplate: Dockerfiles, CI config, scaffolded components. A project setup skill might include templates/docker-compose.yml and templates/.env.example that Cline customizes per project.
  • scripts/ — the secret weapon. Validation, linting, data processing, API health checks. A 500-line linter runs completely outside the LLM and produces a clean "Passed ✓" in context. Zero tokens for the code, just the result.
Use scripts/ when… Use instructions when…
Deterministic operations — linting, validation, formatting Flexible guidance that adapts to context and nuance
Complex calculations — cost estimates, resource sizing Decision-making workflows that depend on situational judgment
Reliability matters — you want consistent, exact behavior every time Best practices and patterns that might vary by team or project
Anything you'd rather not explain — let the script handle it, just show the output Steps that might vary by situation, environment, or preference
Where to put them
◎Project Skills (Team-Shared)
  • Store in .cline/skills/ at your project root (recommended).
  • Also supported: .clinerules/skills/ and .claude/skills/.
  • Commit them to Git. Your whole team benefits — everyone scaffolds with the same patterns, no individual drift.
  • Great for domain-specific skills (your company's infra patterns, project conventions, client-specific workflows).
◎Global Skills (Just for You)
  • macOS/Linux: ~/.cline/skills/
  • Windows: C:\Users\USERNAME\.cline\skills\
  • Available across every project. Good for personal workflows you use everywhere — PR review checklists, your preferred testing patterns, personal deploy helpers.
  • Precedence note: when a global and project skill share the same name, the global skill wins. Keep this in mind when sharing project skills with teammates.
🔁
Toggle, don't delete. Every skill has an on/off switch in the UI. Disable a CI/CD skill when you're doing local dev, re-enable it when you need it. No need to touch the file.
Making skills actually work well
◎Best Practices — Learned the Hard Way
▸
Nail the description first. Everything else is secondary. "Use when deploying to Azure using Bicep, updating ARM templates, or managing Azure resources" will trigger reliably. "Azure helper" will not. Include the exact phrases you'd type when you need this skill.
▸
Offload heavy logic to scripts/. A 300-line validation script is a zero-token operation. Put complexity in scripts, and let the instructions focus on guidance and decision-making. That's the division of labor.
▸
One skill = one domain, no exceptions. Don't cram dotnet-api-scaffold and dotnet-db-migration into one skill. Keep them separate, toggle them independently. When both are active and needed, they'll both load — but only what's relevant to your request.
▸
Keep SKILL.md under 5,000 tokens. If your instructions are getting long, split detailed content into docs/ files and reference them. Cline fetches those docs only when your request needs them — you get depth without paying for it upfront.
▸
Front-load the common cases. Cline reads SKILL.md sequentially. Put the most common workflows first, edge cases and troubleshooting later (or in a docs/ file). Think about what 80% of your requests will look like and optimize for that.
▸
Include real examples, not just abstract steps. "Run the linter" is weaker than "Run scripts/validate.sh — it outputs a per-file report; if exit code is non-zero, list each failing file and the rule violated." Concrete beats abstract every time.
▸
Commit project skills to Git. Store in .cline/skills/ and push. Your team inherits your institutional knowledge without a single meeting. When someone improves a skill, everyone benefits on the next pull.
🛠️
The Bottom Line

Skills transform Cline from a general-purpose assistant into a specialist who knows exactly how your team works. Start with one skill for a task you repeat every week. Test it, tweak the description until it triggers every time, add a script or two. Then watch it compound across your whole team.
