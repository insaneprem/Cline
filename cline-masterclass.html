Cline Skills

Custom Tools &
MCP Extensions

Give Cline specialist knowledge that loads only when needed — no context bloat, no distractions.

What is it?

A Skill is a modular instruction set packaged as a folder with a SKILL.md file. Think of each skill as a role Cline can step into — "data analyst", "AWS deployer", "Go API scaffolder". Unlike rules (which are always active), skills load on-demand: Cline reads their short description on startup, and only loads the full instructions when your request matches.

Skills use progressive loading: skill metadata costs ~100 tokens per skill at startup, the full instructions load only when triggered (under 5k tokens), and bundled resources load only when explicitly referenced.

📸
Screenshot: Cline Skills Panel The VS Code Cline panel showing the Skills tab — scale icon at the bottom-left. Three skills visible with toggle switches: "aws-cdk-deploy" (ON), "data-analysis" (OFF), "go-api-scaffold" (ON). Each skill shows its name and truncated description.
Why Use It?

Without skills, you're either re-explaining your deployment process every session, or loading a massive rules file that burns context on every single Go function you write. With skills, Cline stays sharp and focused — it knows what to load and when.

Key win: A 500-line validation script inside a skill's scripts/ folder doesn't cost you any context tokens — only its output does. You get all the power, none of the bloat.

How to Use It — Step by Step

Scenario: You're building a Go microservice API and want Cline to scaffold handlers, routes, and tests consistently every time.

  1. Enable Skills: Go to Settings → Features → Enable Skills in VS Code Cline settings.
  2. Open Skills Tab: Click the scale icon at the bottom of the Cline panel → switch to "Skills" tab → click "New skill…"
  3. Name your skill: Enter go-api-scaffold (kebab-case, lowercase). Cline creates a directory with a template SKILL.md.
  4. Edit the SKILL.md: Write a specific description and detailed instructions (see example below).
  5. Add supporting files: Create templates/handler.go.tmpl and scripts/validate-routes.sh in the skill folder.
  6. Test the trigger: Ask Cline "scaffold a new user authentication endpoint" — the skill should auto-activate.
---
name: go-api-scaffold
description: Scaffold Go REST API handlers, routes, middleware,
  and tests. Use when creating new API endpoints, adding
  routes to a Go service, or generating handler boilerplate.
---

# Go API Scaffold

When scaffolding a new API handler, always:

## 1. Create the Handler
- Place in internal/handlers/
- Follow the pattern in templates/handler.go.tmpl
- Use the repository pattern for DB access

## 2. Register the Route
- Add to internal/router/router.go
- Group by feature domain (auth, users, orders)

## 3. Write the Test
- Create _test.go file alongside the handler
- Use table-driven tests
- Mock external dependencies

Run validation: bash scripts/validate-routes.sh
📸
Screenshot: Skill Directory Structure in VS Code Explorer The VS Code file explorer showing .cline/skills/go-api-scaffold/ folder expanded, containing: SKILL.md, templates/handler.go.tmpl, scripts/validate-routes.sh
Best Practices for High Efficiency
  • 🎯
    Nail the description — It's the trigger. Be specific: "Use when deploying to AWS using CDK, updating infrastructure stacks, or managing AWS resources." Vague descriptions mean the skill won't fire when you need it.
  • 📁
    Offload weight to scripts/ — Put complex logic (validation, code generation, linting) in scripts/ subdirectory. Only the output enters context. A 300-line linter becomes a single "Passed" or error message.
  • 🔀
    One skill = one domain — Don't build a mega-skill. Keep go-api-scaffold separate from go-db-migration. This lets you toggle them independently and prevents context pollution.
  • 🗂️
    Commit project skills to Git — Store skills in .cline/skills/ and commit them. Your whole team benefits — a C++ competitive programming template skill shared across the team means everyone scaffolds solutions the same way.
  • ⬆️
    Front-load critical info — Cline reads SKILL.md sequentially. Put the most important cases first. Use clear headers like ## Error Handling so Cline can scan efficiently.
02
Cline Rules

Persistent Context
with .clinerules

Define your coding standards once, enforce them everywhere — automatically, silently, consistently.

What is it?

Rules are markdown files that feed persistent instructions to Cline on every request. Instead of re-explaining "use camelCase in Go, use the repository pattern, never modify /legacy" each session, you define it once in a .clinerules/ folder and Cline follows it automatically.

Cline supports rules from multiple sources: .clinerules/ (primary), .cursorrules, .windsurfrules, and AGENTS.md. Workspace rules beat global rules when they conflict.

📸
Screenshot: Rules Panel with Toggles Cline Rules tab showing 4 rule files: coding.md (ON), testing.md (ON), architecture.md (ON), legacy-freeze.md (OFF). Each has a toggle switch and shows its last modified date.
Why Use It?

Imagine you're on a team building a Go + Next.js microservices platform. Without rules, every developer using Cline gets different output — one uses err, another uses e, one uses inline SQL, another uses repositories. Rules enforce team-wide consistency automatically.

Conditional rules are the power-user feature: rules that only activate when you're working in matching files. Your React component rules won't load when you're editing Go backend code. This keeps Cline focused and saves tokens.

How to Use It — Step by Step

Scenario: Setting up a Go backend microservice with strict standards and a Next.js frontend — each with their own rules.

  1. Create the rules folder: At your project root, create .clinerules/ directory.
  2. Open Rules panel: Click the scale icon in Cline → "Rules" tab → "New rule file…" → name it go-backend.
  3. Write your Go rule with a conditional:
---
paths:
  - "services/**"
  - "internal/**"
  - "cmd/**"
---

# Go Backend Guidelines

## Naming
- Variables: camelCase (userID, not user_id)
- Exported: PascalCase only
- Errors: always named err, wrapped with fmt.Errorf("context: %w", err)

## Architecture
- Use repository pattern for all DB access
- Never write raw SQL in handlers — use internal/store/
- Follow error handling pattern in internal/errors/errors.go
- Do NOT modify anything in /legacy (scheduled Q2 removal)

## Testing
- Table-driven tests for all public functions
- Mock DB using interfaces, never real DB in unit tests
  1. Add a Next.js frontend rule: Create .clinerules/nextjs-frontend.md with paths: ["app/**", "components/**"]. This file only loads when editing frontend code.
  2. Add a universal rule: Create .clinerules/universal.md with no frontmatter — it applies to every request (commit messages, docs, everything).
  3. Verify activation: When you edit a Go file, look for "Conditional rules applied: workspace:go-backend.md" notification in Cline.
📸
Screenshot: Conditional Rule Activation Notification A VS Code notification banner at the bottom of the Cline panel reading "Conditional rules applied: workspace:go-backend.md" — triggered automatically when user opens services/auth/handler.go
Best Practices for High Efficiency
  • ✂️
    Be specific, not vague — "Use descriptive names" wastes tokens. "camelCase for Go variables, UPPER_SNAKE_CASE for C++ constants, PascalCase for exported Go types" gives Cline concrete rules to follow.
  • 💡
    Include the "why" — "Don't modify /legacy (scheduled Q2 removal)" helps Cline make smarter decisions in edge cases. It won't just skip the folder — it'll understand the context.
  • 📌
    Point to examples in your codebase — "Follow the pattern in /src/utils/errors.ts" is 10× more powerful than describing the pattern from scratch. Cline will read that file directly.
  • 🗂️
    One concern per file — Split coding.md, testing.md, and architecture.md so you can toggle them independently. Disable the strict testing rule while prototyping, re-enable before review.
  • 🧪
    Test conditional patterns — Add a test rule with your path pattern and "TEST: This should activate for X files." Work on a matching file and verify the notification fires.

Cline Workflows

Multi-Step Task
Automation

Turn your 12-step release ritual into a single slash command. Type /release.md and walk away.

What is it?

Workflows are Markdown files that define a sequence of steps for Cline to execute. You invoke them by typing /filename.md in the chat. The filename becomes the command — deploy.md → /deploy.md.

Steps can be written in plain English (Cline figures out the implementation), or in precise XML tool syntax for guaranteed behavior. You can mix both in the same workflow — use natural language for flexible steps and XML for the ones that must be exact.

📸
Screenshot: Workflow Autocomplete in Cline Chat Cline chat input showing "/" typed, with a dropdown listing available workflows: deploy.md, release-prep.md, setup-db.md, test-and-commit.md — with descriptions next to each.
Why Use It?

Multi-step processes are where engineers waste the most time and make the most mistakes. Deploying a microservice involves checking git status, running tests, building Docker images, updating Kubernetes manifests, tagging, and pushing — in the right order. Miss a step and you're debugging production at midnight.

Workflows encode institutional knowledge. New team member? They run /deploy.md and it just works. Workflows also support MCP tool calls, CLI commands, and approval gates — so Cline pauses for your review at the right moments.

How to Use It — Step by Step

Scenario: Building a workflow to run tests, bump version, and prepare a release for your Next.js + Go microservice platform.

  1. Open Workflows panel: Scale icon → Workflows tab → "New workflow file…" → name it release-prep
  2. Write your workflow file (see example below). Use XML for precise steps, natural language for flexible ones.
  3. Store it: Place in .clinerules/workflows/release-prep.md and commit to Git so your team can use it.
  4. Invoke it: Type /release-prep.md in Cline chat. Cline walks through each step, pausing for your approval at the version bump.
# Release Preparation

Prepare a new release: run tests, build, bump version, generate changelog.

## Step 1: Verify clean working directory
<execute_command>
<command>git status --porcelain</command>
</execute_command>
If there are uncommitted changes, ask whether to stash or abort.

## Step 2: Run all tests
<execute_command>
<command>go test ./... && cd frontend && npm test -- --watchAll=false</command>
</execute_command>
If any tests fail, show the failures and stop the workflow.

## Step 3: Ask for version bump
<ask_followup_question>
<question>What version bump should this release be?</question>
<options>["Patch (x.x.X)", "Minor (x.X.0)", "Major (X.0.0)"]</options>
</ask_followup_question>

## Step 4: Update version files
Update version.go and package.json to the new version.

## Step 5: Generate changelog
<execute_command>
<command>git log --oneline $(git describe --tags --abbrev=0)..HEAD</command>
</execute_command>
Use these commits to write a CHANGELOG.md entry for the new version.
💡 Pro tip: After finishing something you'll repeat, tell Cline: "Create a workflow for the process we just completed." Cline analyzes the conversation and generates the workflow file automatically.
Best Practices for High Efficiency
  • 🛡️
    Include failure handling — Every step that can fail should say what to do: "If tests fail, show the failures and stop." Don't let Cline blunder forward after a broken build.
  • 🔗
    Compose, don't bloat — Keep deploy.md focused on deploying. Create a separate setup-db.md. Chain them in chat rather than building one mega-workflow that does everything.
  • 🎛️
    Use XML for precision, prose for flexibility — XML tool calls guarantee exact behavior. Natural language lets Cline adapt to the situation. Use <execute_command> for the critical commands, prose for "summarize what changed."
  • 🔒
    Review before running — Workflows execute with your full permissions. Always read a workflow (especially from external sources) before invoking it for the first time.

Cline Hooks

Lifecycle Scripts &
Guardrails

Inject determinism into AI's non-determinism. Validate, block, log, and augment at every step of Cline's workflow.

What is it?

Hooks are executable scripts that run at key moments in Cline's task lifecycle. They receive a JSON payload on stdin and return a JSON response on stdout. The response can: allow the operation, block it with an error message, or inject context into the conversation.

Hooks bring determinism to an AI tool — they're the programmatic enforcement layer that rules can describe but can't enforce. Rules say "use TypeScript"; a PreToolUse hook actually blocks creation of .js files.

Hook Type When It Fires Best Used For
TaskStartNew task beginsInject project context, log to analytics
TaskResumeInterrupted task resumesRefresh state, check for upstream changes
TaskCancelTask is cancelledClean up temp files, notify Slack
TaskCompleteTask succeedsRun tests, trigger CI/CD, notify team
PreToolUseBefore any tool executesBlock dangerous ops, validate params
PostToolUseAfter tool completesAudit logs, performance monitoring
UserPromptSubmitYou send a messageAnalytics, prompt enrichment
PreCompactBefore context truncationArchive important conversation parts
Why Use It?

You've told Cline "always use TypeScript" in a rule. But rules are suggestions — Cline might still write a .js file if it misreads context. A PreToolUse hook catches that before it happens and blocks it with a clear message. That's the difference between guidance and enforcement.

Hooks are also the bridge to your existing tooling: run your linter before files are saved, trigger your Slack bot when a task completes, inject environment-specific context when Cline starts working on your staging service vs. production service.

📸
Diagram: Hook Lifecycle Flow A flowchart showing: TaskStart/TaskResume → (UserPromptSubmit → Cline processes) loop → (PreToolUse → Tool Executes → PostToolUse) loop → PreCompact (if needed) → TaskComplete/TaskCancel. Orange nodes highlighted for hook injection points.
How to Use It — Step by Step

Scenario: You want to (1) block .js files in your TypeScript microservice, and (2) auto-inject your project context at task start.

  1. Open Hooks tab: Scale icon → Hooks tab → "New hook…" → select PreToolUse
  2. Review and edit the script (click pencil icon). The hook is created as PreToolUse (extensionless on macOS/Linux, PreToolUse.ps1 on Windows).
  3. Write the TypeScript enforcement hook:
#!/bin/bash
# PreToolUse hook: Block .js files in TypeScript project

INPUT=$(cat)
TOOL=$(echo "$INPUT" | jq -r '.preToolUse.tool')
FILE_PATH=$(echo "$INPUT" | jq -r '.preToolUse.parameters.path // empty')

if [[ "$TOOL" == "write_to_file" && "$FILE_PATH" == *.js ]]; then
  echo '{"cancel":true,"errorMessage":"Use .ts files — this is a TypeScript project"}'
  exit 0
fi

echo '{"cancel":false}'
  1. Make it executable: chmod +x ~/.cline/hooks/PreToolUse (macOS/Linux)
  2. Create a TaskStart hook to inject context from a .project-context file in your repo.
#!/bin/bash
# TaskStart hook: Inject project context

INPUT=$(cat)
WORKSPACE=$(echo "$INPUT" | jq -r '.workspaceRoots[0] // empty')

if [[ -f "$WORKSPACE/.project-context" ]]; then
  CONTEXT=$(cat "$WORKSPACE/.project-context")
  echo "{\"cancel\":false,\"contextModification\":\"$CONTEXT\"}"
else
  echo '{"cancel":false}'
fi
  1. Enable the hooks in the Hooks tab → toggle them ON. Test by asking Cline to create a .js file — it should be blocked with your error message.
⚠️ Always review hook code before enabling. Hooks run automatically and can block operations or execute shell commands. Use stderr (>&2) for debug output, not stdout — stdout is reserved for the JSON response.
Best Practices for High Efficiency
  • ⚡
    Keep hooks fast — Every hook adds latency to Cline's flow. Avoid network calls in PreToolUse hooks. Save async operations (Slack notifications, logging) for PostToolUse or TaskComplete.
  • 📊
    Use PostToolUse for analytics — Log tool name, success, and duration to a local file. Over time you'll see exactly how Cline spends its effort and optimize accordingly.
  • 🔐
    Use hooks as safety nets with auto-approve — Enable auto-approve for speed, then use PreToolUse hooks as the actual enforcement layer. You get fast AI + deterministic guardrails.
  • 🗂️
    Project hooks in version control — Store hooks in .clinerules/hooks/. When global and workspace hooks both exist for the same type, both run (global first). Use this to layer team policies on top of personal preferences.

Cline Memory Bank

Long-Term Context
Across Sessions

Turn Cline from a stateless assistant into a persistent development partner that remembers your entire project.

What is it?

Memory Bank is a documentation methodology — a structured set of Markdown files that give Cline persistent "memory" across sessions. Since every new Cline conversation starts fresh, Memory Bank is how you re-establish context instantly with a single command: "follow your custom instructions."

Files live in a memory-bank/ folder at your project root. They're organized hierarchically — from foundational (project brief) to current (active focus). Cline reads them at session start to rebuild a complete picture of your project.

.md
projectbrief.md
Foundation: core requirements, goals, scope of truth. Created once, rarely changes.
.md
productContext.md
Why the project exists, problems solved, UX goals, target users.
.md
activeContext.md
Current focus, recent changes, next steps. Updates most frequently — after every session.
.md
systemPatterns.md
Architecture, design patterns, component relationships, critical implementation paths.
.md
techContext.md
Tech stack (Go, Next.js, Postgres), dev setup, constraints, tool usage patterns.
.md
progress.md
What works, what's left to build, milestones, known issues.
📸
Screenshot: memory-bank/ folder in VS Code Explorer File explorer showing memory-bank/ folder with all 6 core files listed. projectbrief.md shows "2.1KB", activeContext.md shows "(modified today)", progress.md visible. Context window progress bar in Cline showing 23% used after loading Memory Bank.
Why Use It?

Here's the problem: you spend 3 days building a blog platform with Go microservices. You've made hundreds of architectural decisions — CQRS here, event sourcing there, specific error patterns everywhere. On day 4 you open VS Code and start a new Cline task. All of that context is gone.

With Memory Bank, you end day 3 with "update memory bank" and start day 4 with "follow your custom instructions." Cline reads the files, reconstructs the full picture, and picks up exactly where you left off — architecture, patterns, current focus, and all.

How to Use It — Step by Step

Scenario: Initializing a Memory Bank for a microservices blog platform (Go backend + Next.js frontend + Postgres).

  1. Add the custom instructions: Copy the Memory Bank instructions (from docs) into Cline's custom instructions or a .clinerules file.
  2. Initialize: Tell Cline "initialize memory bank" — it creates the memory-bank/ folder and all 6 core files with template content.
  3. Fill in projectbrief.md with your project's core goals, tech stack, and constraints. This is the one file to invest time in upfront.
  4. Add a .clineignore: This is the single biggest optimization. Exclude node_modules/, vendor/, build artifacts. Drop context from 200k+ tokens to under 50k.
  5. End every significant session: Tell Cline "update memory bank" — it reviews all files and updates what changed.
  6. Start every new session: Tell Cline "follow your custom instructions" — it reads all memory bank files and continues seamlessly.
Key commands to memorize:
"initialize memory bank" — First-time setup
"update memory bank" — Save current state
"follow your custom instructions" — Resume from memory
Best Practices for High Efficiency
  • 📏
    Keep files concise — Each file adds to your context on session start. One page for projectbrief.md, current-only for activeContext.md (not a running log). If a file exceeds two pages, split the detail into a linked doc and let Cline read it on demand.
  • 🔄
    activeContext.md is your daily driver — This file changes most. Update it at the end of every session with "what I was doing, what I just completed, what's next." It's your developer handoff note to future-you.
  • 🎯
    Pair with Plan & Act mode — Start sessions in Plan mode: Cline reads the memory bank and you discuss strategy. Switch to Act mode once aligned. This prevents wasted effort from misaligned assumptions.
  • 🧹
    Use .clineignore aggressively — Add it early. Exclude node_modules/, vendor/, .git/, dist/, build artifacts. Smaller starting context = faster responses + cheaper API calls + better focus.
  • 🔗
    Pair Memory Bank + Checkpoints — Memory Bank preserves your project knowledge; Checkpoints preserve your code state. Together: if you need to roll back code, your Memory Bank still holds the context of why you made those decisions.

Cline Checkpoints

Save States &
Fearless Iteration

Experiment at full speed. Every file change is snapshotted. The cost of a mistake drops to nearly zero.

What is it?

Checkpoints are automatic file snapshots taken after every tool use (file edit, command run, etc.). They're stored in a shadow Git repository — completely separate from your actual Git history. Your commits stay clean; Cline's shadow repo tracks everything behind the scenes.

After each tool use, a checkpoint indicator appears in your conversation with Compare and Restore buttons. You can view exactly what changed and restore to any point — keeping or discarding the conversation history independently from the code changes.

📸
Screenshot: Checkpoint Indicator in Cline Chat A Cline conversation showing Cline editing 3 files. After each edit, a small bookmark icon labeled "Checkpoint" appears with a dotted connector line, "Compare" and "Restore" buttons. The third checkpoint is highlighted — user is about to click "Compare."
Why Use It?

Without checkpoints, letting Cline run with auto-approve feels like skydiving without a parachute. It might make 15 file changes before you notice the approach is wrong. You're stuck manually reverting or digging through git diff.

With checkpoints enabled, you can let Cline work fast and autonomously. If it goes sideways, you roll back to the last good state in one click — without losing the conversation context that got you there. The iteration loop collapses from minutes to seconds.

How to Use It — Step by Step

Scenario: Asking Cline to refactor a Go service's database layer. You want to experiment freely but have an escape hatch.

  1. Verify checkpoints are on: They're enabled by default. Check via Cline settings → "Feature Settings" → "Enable Checkpoints."
  2. Enable auto-approve for file edits — checkpoints make this safe. Cline will work quickly without pausing for every change.
  3. Let Cline run the refactor. Watch the checkpoint markers appear after each edit in the chat.
  4. Click "Compare" on any checkpoint to open a diff view — see exactly what changed, across all modified files, before deciding to keep it.
  5. If something's wrong, click "Restore" and choose your restoration strategy (see options below).
  6. Give more specific guidance and let Cline try again — the conversation context is preserved so you don't start from scratch.
Restore Files

Reverts code to this snapshot. Keeps the conversation. Best when: code broke but the discussion was productive.

Restore Task Only

Deletes messages after this point. Keeps the code. Best when: code is good but you want to re-ask the question differently.

Restore Files & Task

Resets both code and conversation. A full restart from a known-good state. Best for: complete do-overs.

📸
Screenshot: Checkpoint Restore Menu The Restore dropdown open in Cline, showing three options: "Restore Files", "Restore Task Only", "Restore Files & Task" — each with a brief description. The "Restore Files" option is highlighted on hover.
Best Practices for High Efficiency
  • 🚀
    Checkpoints unlock auto-approve — This is the key insight. Enable auto-approve for speed, rely on checkpoints for safety. Review the final result, restore if it went wrong. This approach is faster than reviewing every change manually.
  • 🔬
    Use Compare before committing to Git — Before running git add, click Compare on the final checkpoint. It opens a full diff in your editor's native diff viewer — a clean review of everything Cline touched.
  • ⚡
    On large repos, watch for slowdown — Checkpoints write to a shadow Git repo after every tool use. On massive monorepos (100k+ files), this can be slow. Consider disabling for read-heavy exploration tasks and re-enabling for write-heavy work.
  • 🧪
    Use for risky experiments — Want to try a completely different architecture approach? Let Cline go for it. Worst case: one click to restore. Best case: it worked. Low downside, high upside.
  • 🗂️
    Pair with Memory Bank — Memory Bank gives you the "why", Checkpoints give you the "what." If you restore code to an earlier checkpoint, your Memory Bank still has the context of what you were trying to achieve and the decisions you made along the way.
