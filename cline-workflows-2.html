How It Works

One command.
Infinite consistency.

Workflows are Markdown files that guide Cline through any process — with your approval at every stage.

01
📝
Write a Markdown File

Author a .md file with a title and numbered steps. Plain English, XML tools, or CLI — your choice.

02
⚡
Invoke with /name.md

Type / in Cline's chat. Live autocomplete shows all workflows. Select, press Enter — begins immediately.

03
🤖
Cline Executes Step by Step

Each step runs in sequence with approval checkpoints. Reject any step to stop immediately. You stay in control.

04
♻️
Reuse, Share, Evolve

Commit to git, share with your team, review in PRs. Toggle on/off without deleting. Automation lives forever.

What Goes Inside Workflows

Four languages.
Endless flexibility.

Mix and match. Start with plain English, tighten with XML where precision matters.

Natural Language
Plain English Instructions

Write steps as you'd explain them to a colleague. Cline picks the right tools automatically — great for context-aware, adaptive automation.

No special syntax — just describe the goal
Cline adapts intelligently to context changes
Great for conditional and branching logic
natural-steps.md
## Step 1: Check for changes Look at the git status. If there are uncommitted changes, ask the user whether to continue or abort. ## Step 2: Run all tests Execute all tests. If any fail, show the failures and stop. ## Step 3: Write release summary Summarise changes in plain English for non-technical stakeholders.
Cline XML Tools
Precise Tool Calls

Use XML to invoke Cline's built-in tools directly — deterministic execution with exact parameters. Use when reliability matters most.

Deterministic — executes exactly as written
Control approval requirement per step
Best for production-critical workflows
xml-tools.md
<execute_command> <command>npm run test</command> <requires_approval>false</requires_approval> </execute_command> <ask_followup_question> <question>Deploy target?</question> <options> ["Production","Staging"] </options> </ask_followup_question>
CLI Commands
Shell & Terminal Tools

Reference any tool on your machine — git, npm, docker, gh, make, curl. If your terminal can run it, your workflow can orchestrate it.

Full access to your entire local toolchain
Chain multiple CLI tools in sequence
Use output of one step as input to the next
cli-steps.md
## Step 3: Generate changelog git log --oneline \ $(git describe --tags --abbrev=0)..HEAD ## Step 4: Docker build & push docker build -t myapp:latest . docker push registry/myapp:latest ## Step 5: GitHub release gh release create v2.4.0 --generate-notes
MCP Tools
External Service Integrations

Connect to GitHub, Slack, Jira, databases — any MCP server you've configured. Natural language or explicit XML for full control.

GitHub, Slack, Jira, Notion and more
Chain calls across multiple services in one flow
Fully automated cross-service workflows
mcp-tools.md
<use_mcp_tool> <server_name>github-server</server_name> <tool_name>create_release</tool_name> <arguments>{ "tag": "v2.4.0", "body": "Changelog here" }</arguments> </use_mcp_tool> ## Notify team Post release note to #engineering
Creating Your First Workflow

From idea to automation
in five minutes.

1

Open the Workflows Menu

Click the scale icon at the bottom of the Cline sidebar, left of the model selector. Switch to the Workflows tab.

2

Create a New Workflow File

Click "New workflow file…" and enter a name like deploy. Cline appends .md automatically.

3

Write Your Steps

Add a title and numbered steps in Markdown. Be as high-level or specific as needed — Cline adapts intelligently.

✦

Pro Tip: Let Cline Write It

After any repeatable task, say "Create a workflow for what I just did." Cline reads the conversation and generates the file automatically.

5

Invoke with /filename.md

Type / in chat, pick from autocomplete, press Enter. Execution starts immediately.

my-first-workflow.md
# My First Workflow Sets up a fresh feature branch and checks all prerequisites are met. ## Step 1: Check git status Run git status and confirm the working directory is clean. ## Step 2: Create feature branch Ask for branch name, then run git checkout -b [branch-name]. ## Step 3: Install dependencies Run npm install and report any security vulnerabilities found. ## Step 4: Start dev server Start npm run dev and confirm the app loads at localhost.

Best Practices

Write workflows like a
senior engineer.

Six habits that separate professional automation from brittle one-off scripts.

🌱
Start Simple

Write natural language steps first. Only add XML tool syntax when you need guaranteed, deterministic behaviour — not before.

🎯
Keep Workflows Focused

deploy.md deploys. setup-db.md sets up the database. Split complex processes into small, composable files.

❓
Be Explicit About Decisions

When a step needs user input, say so: "Ask whether to deploy to production or staging before continuing."

🛡️
Handle Failure Explicitly

Tell Cline what to do when things break: "If tests fail, show full output and stop the workflow."

📦
Version Control Everything

Store in .clinerules/workflows/ and commit. Review in PRs. Treat them like production code — because they are.

🔄
Automate Your History

After any repeatable task, tell Cline: "Create a workflow for what I just did." Your context becomes instant automation.

Real World Example

Release Preparation
Workflow

A complete workflow combining XML tool calls and natural language — one command for the full release process.

release-prep.md invoke: /release-prep.md
# Release Preparation Prepare a new release: run tests, build, bump version, generate changelog. ## Step 1: Check for clean working directory <execute_command> <command>git status --porcelain</command> </execute_command> If uncommitted changes exist, ask whether to continue or stash them first. ## Step 2: Run the test suite <execute_command> <command>npm run test</command> </execute_command> If any tests fail, stop the workflow and show the full failure output. ## Step 3: Ask for new version <ask_followup_question> <question>What should the new version be?</question> <options>["Patch (x.x.X)", "Minor (x.X.0)", "Major (X.0.0)"]</options> </ask_followup_question> ## Step 4: Generate changelog <execute_command> <command>git log --oneline $(git describe --tags --abbrev=0)..HEAD</command> </execute_command> Use these commits to write a changelog entry and update CHANGELOG.md. ## Step 5: Commit, tag, and push Run git add, commit as "chore: release vX.X.X", tag, and push with tags to origin.
