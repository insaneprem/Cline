What & Why
◎What exactly is a Skill?
  • A Skill is just a folder with a SKILL.md file inside — think of it as a specialist role Cline steps into on demand: tizen-c-review, tizen-cpp-review, tizen-csharp-nui. Nothing magic — just structured knowledge that loads when you need it.
  • Unlike rules (which are always-on and always burning context), Skills activate only when your request matches their description. Everything else stays completely out of the way.
  • Bundled docs/ and templates/ load only when Cline actually needs them — so you're never paying token cost for context you didn't ask for.
◎Why bother with Skills?
  • Here's what happens without Skills: you either re-explain your Tizen review standards on every single session, or you cram everything into a rules file that fires unconditionally — whether you're doing a C memory audit or just tweaking a string constant.
  • Skills keep Cline genuinely focused — Tizen C memory rules stay out of C# NUI sessions, and C# NUI patterns don't leak into bare-metal C reviews. The right context, at the right time.
  • One SKILL.md per domain. Toggle independently. You get a specialist, not a generalist — exactly when you need one, silent when you don't.
🎯 The real payoff: once you write the SKILL.md once, you never explain those standards again — to Cline or to any teammate who pulls your repo.
Anatomy of a Skill
◎What Goes Inside a Skill Directory

Every Skill is literally just a folder. SKILL.md is the only file Cline actually requires — it holds both the trigger description and your core review instructions. Everything else is optional support material that Cline pulls in only when relevant.

SKILL.md — structure overview 
---
name:        my-skill       # must match the directory name exactly
description: Brief, specific description of what this does and WHEN to use it.
             Max 1,024 characters. This is your trigger phrase.
---

# My Skill

Detailed instructions that Cline follows when this skill is activated.
Put the most common cases first — Cline reads sequentially.

Creating your first skill
◎Step-by-Step: Your First Skill

Scenario: You're a Tizen code reviewer. You're tired of pasting the same C memory rules into every chat. You want Cline to enforce C memory safety, C++ EFL patterns, and C# NUI standards — automatically, without re-explaining anything.

1
Open the Skills panel
Click the scale icon at the bottom of the Cline panel → Skills tab → "New skill…" → name it tizen-code-review.
2
Write your SKILL.md
Add YAML frontmatter with a precise description in your SKILL.md. Include the exact phrases you'd naturally type: "review this Tizen C file", "audit the NUI code", "check my TIDL interface". Then route each language to its own doc file so SKILL.md stays lean.
3
Add language-specific review docs
Create docs/c-native-review.md (memory, privilege APIs), docs/cpp-review.md (RAII, EFL), docs/csharp-nui-review.md (View lifecycle, async). Cline loads only the one relevant to the current file.
4
Test: does it trigger?
Say "review this Tizen C file for memory leaks". You should see "Activating skill: tizen-code-review" in the Cline panel. If it doesn't trigger — don't overthink it, just add more of your own natural phrases to the description in SKILL.md and try again.
docs/c-native-review.md — Tizen C Review 
# Tizen Native C — Review Checklist

## Memory Safety
- Every malloc() / calloc() must have a matching free() on ALL exit paths
- Prefer g_malloc() / g_free() for GLib-managed objects
- Flag unbounded strcpy(), sprintf() — require strncpy() / snprintf()

## Privilege & API Usage
- Cross-check every app_control_*(), location_*() call against tizen-manifest.xml
- http://tizen.org/privilege/network.get required for any socket call
- No direct syscall() — use Tizen CAPI wrappers only

## Error Handling
- Every Tizen CAPI call returns int — check against TIZEN_ERROR_NONE
- Use dlog_print() with correct log level; no bare printf() in production
docs/cpp-review.md — Tizen C++ / EFL Review 
# Tizen C++ / EFL — Review Checklist

## Object Lifecycle (RAII)
- Evas objects via elm_*() are parent-owned — do NOT delete them manually
- Eina containers: always call eina_list_free() with a data-free callback
- Pair every evas_object_smart_callback_add() with a matching del callback

## Threading
- UI updates MUST run on the main thread — wrap with ecore_main_loop_thread_safe_call_async()
- Flag any std::thread that touches Evas_Object directly

## TIDL Interface
- Verify all .tidl type definitions match the implementation — flag any mismatch
docs/csharp-nui-review.md — Tizen C# NUI Review 
# Tizen C# NUI — Review Checklist

## View Lifecycle
- Override Dispose(bool) for any View that holds unmanaged resources
- Animation objects: call .Stop() and .Dispose() before removing associated views
- Do NOT hold strong references to Window.Instance in long-lived objects

## Async / Await
- All async methods touching NUI must resume on the UI thread
- Flag async void (except event handlers) — require async Task

## Data Binding
- Confirm INotifyPropertyChanged on all bound ViewModels
- Two-way bindings on TextField: BindingMode.TwoWay must be explicit

## Privilege
- Match tizen-manifest.xml against all Tizen.Security.PrivacyPrivilegeManager calls

Docs, templates, and support files
◎Bundled Supporting Files — What Goes Where

Skills carry a full folder of resources alongside SKILL.md. Cline loads each file only when it's actually referenced — so you can bundle as much detail as you want without paying for it on every session.

  • docs/ — content too detailed for SKILL.md, or only relevant in specific situations. The review skill routes to the right language doc automatically. You never have to tell Cline which one to open.
  • templates/ — review report boilerplate, TIDL stubs, manifest snippets. A templates/review-report.md gives every review a consistent structure with zero manual setup on your part.
  • Keep SKILL.md under 5,000 words. Route the heavy detail into docs/ files. Your SKILL.md stays fast and scannable — the rest only loads when it's actually needed.
Put in docs/ when… Put directly in SKILL.md when…
Language-specific detail — C memory rules, C# NUI lifecycle, EFL threading patterns Universal rules that apply to all Tizen code regardless of language
Platform variants — mobile vs TV vs wearable specific guidelines Routing logic — which doc to load based on the file type under review
Deep checklists that would make SKILL.md too long to scan Trigger phrases and quick-reference steps used on every review
Where to put them
◎Project Skills — Team-Shared
  • Store in .cline/skills/ at your project root. That's the only path you really need to remember.
  • Commit to Git. Every reviewer picks up the same Tizen standards automatically. When someone tightens the C# checklist in SKILL.md, the whole team gets it on the next pull — no Slack message required.
  • Also works with: .clinerules/skills/ and .claude/skills/ if your project already uses those conventions.
◎Global Skills — Personal
  • macOS / Linux: ~/.cline/skills/
  • Windows: C:\Users\USERNAME\.cline\skills\
  • Available across every project on your machine — handy for personal review preferences or opinionated defaults that aren't really the team's business.
  • Heads up: when a global Skill and a project Skill share the same name, the global one wins. Worth knowing before you wonder why the team version isn't loading.
🔁
Toggle, don't delete. Every Skill has an on/off switch in the panel. Disable the C++ review SKILL.md during a pure C# sprint — flip it back on when you need it. No file editing, no context menu hunting.
Making skills actually work well
◎Best Practices — Learned the Hard Way
▸
Nail the description in your SKILL.md first. "Review Tizen C or C++ code for memory leaks, privilege misuse, or TIDL errors" fires reliably every time. "Tizen helper" never fires. Write it the way you'd actually type the request — Cline matches on your words, not on ideal phrasing.
▸
Route, don't cram. Your SKILL.md should route to the right doc — not contain every rule itself. docs/c-native-review.md handles C. docs/csharp-nui-review.md handles C#. Cline reads only whichever one actually applies, so neither bleeds into the other's session.
▸
One Skill = one domain. Keep tizen-c-review and tizen-csharp-nui as separate folders with separate SKILL.md files. Toggle them independently. When a request genuinely needs both, both load — otherwise they stay completely silent and out of context.
▸
Front-load the common cases in your SKILL.md. Cline reads it top to bottom. Put the C native path first — it's the most common Tizen code path. Wearable-specific edge cases go last, or better yet, move them into a docs/ file entirely.
▸
Be concrete, not abstract. "Check tizen-manifest.xml for every privilege the code calls" gets followed. "Check privileges" gets interpreted loosely. The more specific your SKILL.md instructions are, the more reliably Cline acts on them.
▸
Commit your Skills to Git — seriously. Store in .cline/skills/ and push. Every reviewer on the team inherits the same standards from that moment on. No onboarding doc, no Slack thread, no meeting. Just a pull.

The Bottom Line

Skills turn Cline into a genuine Tizen specialist — C memory rules, C++ EFL patterns, C# NUI lifecycle — on-demand, exactly when you need them, silent when you don't. Start small: one SKILL.md for your most-repeated review type. Tweak the description until it triggers every time. Then add the language docs. Commit it to Git and let the whole team benefit — one pull and everyone's on the same standards.
